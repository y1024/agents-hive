# Security Model: Minimum HITL + SafeExecutor-first

> Status: **Shipped** (Phase 1 of `add-spec-driven-cognition`, 2026-04).
> Rollback knob: `security.permission_mode = "strict"`.

## 目标

- **最小化 UI 摩擦**：Web/IM 双通道，常规命令不弹权限窗。
- **红线不可逾越**：`rm -rf /` / `mkfs.*` / `DROP TABLE` 等 19 条内置模式（`internal/security/builtin_rules.go:BuiltinDangerousRules`）无论会话类型全部 Deny。
- **IM 通道无 UI**：PolicyAsk 在 IM 上自动放行（有 audit log + metric），PolicyDeny 仍必须拒绝。
- **一键回滚**：`strict` 模式把所有 shell + 非 shell 工具全部推进 HITL 分支。

## 决策路径（`internal/master/lifecycle.go`）

```
createPermissionPromptFn(ctx, req)
  │
  ├─ 非 shell 家族 tool（bash/shell/exec/run_command 之外）
  │    ├─ minimal ⇒ {Granted: true}
  │    └─ strict  ⇒ HITL
  │
  └─ shell 家族 tool
       ├─ Input JSON 无 "command" / 空 ⇒ safe-deny + warn
       └─ MatchPolicy(cmd):
            ├─ PolicyDeny  ⇒ {Granted: false} + metric security.policy_deny_total{pattern}
            ├─ PolicyAsk + IM session + minimal ⇒ {Granted: true} + warn + metric security.policy_ask_im_autoallow_total
            ├─ PolicyAsk + (非 IM 或 strict)    ⇒ HITL
            └─ PolicyAllow / 未匹配
                 ├─ minimal ⇒ {Granted: true}
                 └─ strict  ⇒ HITL
```

**关键不变量**：`MatchPolicy` **严格早于** `strings.HasPrefix(sessionID, "im-")` 判定。任何顺序调换都会让 IM 用户能打穿 `rm -rf /`。

## 配置

`config.json`（或等价环境变量）：

```json
{
  "security": {
    "permission_mode": "minimal",
    "destructive_patterns": [
      {
        "pattern": "^systemctl\\s+stop\\s+postgres",
        "policy": "deny",
        "description": "禁止直接停生产 Postgres"
      }
    ]
  }
}
```

- `permission_mode`: `minimal`（默认）或 `strict`。
- `destructive_patterns`: **追加**到 `security.BuiltinDangerousRules` 之后，不覆盖内置规则。
- 热重载：`Master.UpdateSecurityConfig` 通过 `atomic.Pointer[SafeExecutor]` 原子替换，零锁读。

## Metrics

所有指标由 `m.enqueueMetric` 旁路投递，无需预注册。

| 指标 | 标签 | 含义 | 报警阈值 |
|------|------|------|---------|
| `security.policy_deny_total` | `tool` / `session_id` / `im` / `pattern` | PolicyDeny 触发次数；`pattern` = 命中规则的正则或 `ast:dangerous` | 单小时 > 0 立即人工介入 |
| `security.policy_ask_im_autoallow_total` | `tool` / `session_id` | IM + PolicyAsk 自动放行次数 | 单 session / 天 > 10 触发 `security.policy_ask_im_autoallow_rate_exceeded` |
| `security.shell_input_malformed_total` | `tool` / `session_id` | shell 家族工具 Input JSON 解析失败 safe-deny 次数 | 突发 > 5/min 排查上游调用方 |
| `security.safe_executor_missing_total` | `tool` / `session_id` | `safeExecutor` 未初始化触发 fail-closed（理论不可达） | 任意触发即告警（代码 invariant 破坏信号） |

## Runbook

**`policy_ask_im_autoallow_rate_exceeded` 触发时**：
1. 找到对应 `session_id`，拉审计日志过滤命令分布。
2. 判断是否是合法场景（运维批量 `rm -rf /tmp/*`）还是潜在滥用。
3. 必要时对该 session 强制回退 `strict` 模式直到复盘结束。

**`policy_deny_total` 触发时**：
1. 按 `pattern` label 分组，定位命中哪条规则（正则 or `ast:dangerous`）。
2. 结合 audit log 的 `session_id` / `tool` / `command` 字段看上下文。
3. 如果是误杀 → 加 `destructive_patterns` 的 `policy: ask` 条目覆盖。
4. 如果是真实攻击 → 审计 IM / Web token 泄露链路。

**`safe_executor_missing_total` 触发时**：
代码 invariant 已被破坏（`NewMaster` 未运行 security 初始化即对外提供服务）。立即 P0，检查 bootstrap 启动日志，考虑紧急回滚至上一版本。

## 回滚

单行配置：`security.permission_mode = "strict"`。效果：

- 所有 shell + 非 shell 工具一律走 HITL 分支。
- IM session 将收到权限请求广播（由于 IM 无 UI，实际表现为 5 分钟后 timeout Deny）。

对业务 input_request（choice_type_registry 驱动的决策流）**不影响** —— 两条路径完全正交。
