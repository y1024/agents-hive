# 调研结论

> 上级索引：[2026-05-16-PPT生成工具与HTML预览实施计划.md](../2026-05-16-PPT生成工具与HTML预览实施计划.md)

## 2. 调研结论

### 2.1 本仓库现状

已有能力：

- `frontend/src/components/canvas/CanvasPanel.tsx` 已有右侧 Canvas，并支持 HTML artifact 预览。
- `frontend/src/components/canvas/renderers/HtmlRenderer.tsx` 使用 iframe `srcDoc` + sandbox 预览 HTML。
- `frontend/src/utils/artifactParser.ts` 已支持 `<artifact type="html">...</artifact>` 和 `<artifact type="ppt">...</artifact>`。
- `frontend/src/store/canvas.ts` 的 `ArtifactType` 已包含 `mermaid`，下载扩展名为 `.mmd`，但当前 preview 路由没有 Mermaid renderer。
- `frontend/src/components/chat/MermaidBlock.tsx` 已能在聊天正文中懒加载 Mermaid 渲染代码块，使用 `securityLevel:'strict'` 并做基础 SVG sanitize。
- `frontend/src/components/chat/BusinessCodeRenderer.tsx` 对 `language=mermaid` 直接返回 `MermaidBlock`，但不会打开 Canvas。
- `frontend/src/components/chat/ArtifactCard.tsx` 已有 PPT 图标卡片，但 `ppt` 当前实际按 Markdown 渲染。
- `internal/asset` 已有统一对象存储服务，支持上传资产并返回 `asset://...`。
- `internal/api/asset_handlers.go` 已有 asset resolve/proxy 下载路径。
- `internal/router/capability_registry.go` 已有 `generate_image` / `generate_video` 这类 `media + local_write` 内置工具画像，可照此注册 `generate_ppt`。
- `internal/fileconv/office.go` 能从 PPTX 提取文本，适合作为生成后验收测试的一部分。

关键缺口：

- 没有结构化 presentation 数据模型。
- 没有 PPTX 生成器。
- 没有 `generate_ppt` 工具。
- Canvas 没有“presentation artifact + PPTX 下载资产”的元数据模型。
- Canvas 没有 Mermaid / mindmap 专用 preview renderer、下载源 + SVG/PNG 的 asset 元数据模型。
- `artifactParser.ts` 还不识别 `<artifact type="mermaid">`，`ArtifactManifest` 也没有 Mermaid/mindmap 类型。
- 没有 `generate_diagram` / `generate_mindmap` 工具，脑图和 Mermaid 只能由模型输出代码块，无法服务端稳定导出图片。
- 当前 HTML 预览默认禁用 scripts；slide deck 翻页需要受控脚本执行。

### 2.2 `guizang-ppt-skill` 源码结论

本地路径：`/Users/guoss/workspace/guizang-ppt-skill`

可复用内容：

- `assets/template.html`：Style A，电子杂志/电子墨水 HTML deck 模板。
- `assets/template-swiss.html`：Style B，瑞士国际主义 HTML deck 模板。
- `references/layouts.md`：Style A 版式骨架。
- `references/layouts-swiss.md`：Style B 22 个锁定版式。
- `references/themes.md` / `references/themes-swiss.md`：主题色约束。
- `references/checklist.md`：质量检查清单。
- `scripts/validate-swiss-deck.mjs`：Swiss 版式校验器。
- `LICENSE`：MIT License，允许复用，但必须保留版权和许可声明。

不应直接复用的点：

- 不接入为 Hive skill。
- 不让模型直接自由拼完整 HTML 后再转 PPTX。
- 不照搬外部 CDN 依赖。Hive 里应尽量使用自包含 CSS/JS，避免生成的预览 iframe 对 Google Fonts、unpkg 等外部资源产生运行时依赖。

源码级借鉴点：

- `scripts/validate-swiss-deck.mjs` 用正则提取 `<section class="slide">`，检查 `data-layout` 是否来自 `S01`-`S22` 白名单，禁止未登记的 `P23/P24` 结构，禁止正文页顶部标题居中，禁止 SVG 可见 `<text>`，要求本地图片带 `data-image-slot`，并对 `S15/S16/S22` 图片槽位做比例和裁切规则校验。Hive 应把这些规则前移到 `DeckSpec` 校验，同时在 HTML/PPTX renderer 输出后做一次结构验收。
- `references/swiss-layout-lock.md` 是比模板更重要的“版式契约”。它把每个 Swiss layout 的骨架、图片规则和禁用项写清楚，适合转成 `internal/presentation/spec/layouts.go` 中的 layout registry。
- `assets/template-swiss.html` 的核心价值是设计 token 和组件系统：主题 CSS 变量、字体栈、Carbon 2x spacing、chrome/footer、KPI、卡片、timeline、图片 frame、翻页脚本。落地时应抽取 token 和结构，不应直接复制整份带外链字体/CDN/WebGL 动效的 HTML。
- `references/layouts-swiss.md` 和 `references/layouts.md` 适合用作 HTML renderer 的骨架来源，并反向定义 PPTX 坐标映射；不要让模型临时创造自由 HTML 结构。

### 2.3 `html-anything` 源码结论

本地路径：`/Users/guoss/workspace/html-anything`

许可证：Apache-2.0。可以借鉴和复用思想；如复制具体源码片段，需保留许可证归属。

核心发现：

- `src/lib/extract-html.ts` 解决了 agent 输出“带解释文字 / markdown fence / 不完整 HTML”的问题。它会从输出中提取真实 HTML，并在预览阶段补上 `</body></html>`，让 iframe 更稳地渲染。Hive 的普通 HTML artifact 可借鉴这个容错逻辑；`generate_ppt` 工具路径首发不做流式预览，应返回结构化 JSON 和 asset，不依赖模型直接吐大段 HTML。
- `src/lib/deck.ts` 采用统一约定：每页是顶层 `<section class="slide" data-slide-id="N">`，speaker notes 放在 `<aside class="notes">`。它把每页拆成独立 standalone HTML 文档，复制原 `<head>`、`body class/style`，并隐藏 notes。Hive 的 `DeckRenderer` 应采用类似的 per-slide iframe 模型，但对受信工具输出使用更严格 sandbox。
- `src/components/deck-viewer.tsx` 已实现 deck 专用预览体验：主 iframe、上一页/下一页、键盘导航、全屏演示、notes 面板、底部缩略图条。Hive 右侧 Canvas 可以借鉴这个交互结构，比单个全页 HTML iframe 更适合 PPT。
- `src/lib/export/image.ts` 的截图逻辑值得保留为视觉兜底和 QA：等待 iframe document complete、stylesheet、fonts、images decode 和 layout settle，再按真实 layout width/height 截图，避免中文换行和滚动区域裁切问题。Hive 主导出不走截图；visual PPTX 首选服务端 Playwright，浏览器端截图只作为 QA/PNG zip 辅助。
- `src/lib/export/deck.ts` 的 PPTX 导出是每页 screenshot PNG 全铺到 PPTX slide，并把 notes 写入 `slide.addNotes`。这不是可编辑 PPTX，只能作为 M3 “视觉版 PPTX”内部兜底或后续显式能力，首发不进入用户 UI。
- `src/components/export-menu.tsx` 的导出菜单把普通 HTML 下载、PNG 截图、deck PDF、deck PNG zip、deck PPTX 聚合在一处。Hive 首发应简化为“预览”和“下载可编辑 PPTX”；HTML 下载可作为二级入口，视觉版导出不公开。
- `src/lib/templates/loader.ts` 使用 folder-per-template registry：每个模板目录包含 `SKILL.md`、`example.md`、`example.html`，通过 frontmatter 提供 picker 元数据和示例预览。Hive 不需要把它做成用户安装 skill，但可以把 `presentation/templates/<id>` 做成后端可枚举 registry，方便新增视觉系统和 golden fixture。
- `src/app/api/convert/route.ts` 和 `src/lib/use-convert.ts` 的 diff-edit 思路可用于二期“修改这份 PPT”：保存 `baseContent/baseHtml`，下一次只让模型做最小差异编辑。Hive 更好的版本是保存 `DeckSpec`，对 DeckSpec 做局部修改，再重新渲染 HTML/PPTX。

不应直接搬的点：

- 不把 agent CLI spawn/SSE convert 流程搬进 Hive。Hive 已有 chat、tool call、asset 和权限体系，PPT 生成应是内置 tool/service。
- 不把 `sandbox="allow-same-origin allow-scripts"` 用在普通外部 HTML 上。首发 presentation 预览不执行 iframe 脚本；后续如果受信工具 deck 确需脚本，也只能允许固定内联脚本且不加 `allow-same-origin`。截图路径应在服务端 Playwright 或专门受控 renderer 里执行。
- 不把 `html-anything` 的 screenshot PPTX 作为默认 PPTX。默认必须是 PptxGenJS 原生对象，截图 PPTX 只能标注为“视觉版，不可编辑文本”。
- 不沿用它依赖 Tailwind Play CDN / Google Fonts 的生成要求。Hive 服务端批量生成要尽量自包含、可离线、可复现。

### 2.4 2026 年 PPTX 工具调研

当前调研时间：2026-05-16。

候选工具：

- **PptxGenJS 4.0.1**：MIT。官方支持 Node、React、Vite、浏览器；支持文本、形状、图片、表格、图表、Slide Masters、OOXML PPTX 输出。适合作为可编辑 PPTX 生成内核。
- **Marp CLI 4.4.0**：MIT。适合 Markdown -> HTML/PDF/PPTX，但原生 PPTX 主要偏预渲染；官方讨论里可编辑 PPTX 是实验能力，且依赖 LibreOffice Impress，不适合作为首选生产路径。
- **Slidev 52.15.2**：MIT。官方支持 `slidev export --format pptx`，但文档明确 PPTX 中 slides 会导出为图片，文本不可选择；适合作为演示型输出，不适合作为“可编辑 PPTX”核心。
- **dom-to-pptx / HTML-DOM-to-PPTX 类项目**：目标是把 DOM 映射到 PPTX 对象，方向值得关注，但生态成熟度、CSS 覆盖范围、复杂中文排版和服务端批量稳定性还需要单独 PoC。二期可以评估用于有限 DOM 子集，不应替代首发的 `DeckSpec -> PptxGenJS` 明确映射。

结论：

没有必要找一个单体开源项目覆盖全链路；更好的方案是组合成熟部件：

- `guizang-ppt-skill`：视觉系统、layout registry、校验规则。
- `html-anything`：HTML 抽取、实时预览、deck viewer、缩略图/notes、截图导出兜底。
- `PptxGenJS`：可编辑 PPTX 对象 API。
- Hive 自己：工具调用、权限、安全 sandbox、asset 存储、下载和服务端批量任务。

主路径推荐使用 **PptxGenJS** 的对象 API，而不是 HTML-to-PPTX。PptxGenJS 的 HTML-to-PPTX 文档重点是 HTML table 到 PPTX 表格，不是完整现代 CSS slide deck 转换。若用 HTML 截图或 Slidev/Marp 默认 PPTX 导出，视觉还原高但不可编辑；这可以作为 visual fallback，但不能作为默认交付。

### 2.5 2026 年 Mermaid / 脑图工具调研

当前调研时间：2026-05-16。

候选工具：

- **Mermaid 11.15.0**：MIT。当前前端已依赖 Mermaid 11.x，可用于聊天正文和 Canvas 的客户端预览。需要继续使用 `securityLevel:'strict'`，并对 SVG 输出做 sanitize。
- **@mermaid-js/mermaid-cli 11.15.0**：MIT。适合服务端把 Mermaid 源码导出为 SVG/PNG/PDF；依赖 Playwright/Chromium，部署上应复用 presentation visual worker 的浏览器能力。
- **markmap-lib / markmap-view 0.18.12**：MIT。适合 Markdown -> mindmap 数据结构和浏览器交互渲染；服务端可用 markmap-lib 生成结构，前端用 markmap-view 预览，SVG/PNG 导出可用 Playwright。
- **纯 JSON tree mindmap**：自研 `MindMapSpec`，节点、边、折叠状态、样式 token 都结构化。适合后续编辑和权限审计，比直接让模型吐 markmap HTML 更稳。

结论：

- Mermaid 和脑图不应该塞进 `generate_ppt`，也不应该只作为 Markdown 代码块存在。
- 首发应新增统一 `generate_diagram` 工具，支持 `kind:"mermaid"` 和 `kind:"mindmap"`，返回 `VisualizationArtifact`。
- Mermaid 源码和 MindMapSpec 是事实源；SVG/PNG/HTML 只是渲染产物。
- 客户端可即时预览 Mermaid；服务端导出 SVG/PNG 要走受控 worker，避免浏览器端下载结果与后端审计脱节。

### 2.6 2026 年扩展 Artifact 工具调研

当前调研时间：2026-05-16。

数据图表：

- **Vega-Lite 6.4.3**：BSD-3-Clause。声明式 JSON grammar，天然适合作为 `ChartSpec` 的基础；适合 bar/line/area/scatter/heatmap 等标准统计图，利于 schema 校验、审计和可重复导出。
- **Vega-Embed 7.1.0**：BSD-3-Clause。适合前端 Canvas 嵌入 Vega/Vega-Lite spec，提供一致预览。
- **ECharts 6.0.0**：Apache-2.0。适合业务大屏、关系图、桑基图、雷达图、复杂交互图表。作为 M4 增强，不作为 ChartSpec 首发唯一基础，避免 option schema 过宽。
- **Observable Plot 0.6.17**：ISC。API 体验好，适合开发者手写图；但模型生成和审计更适合 Vega-Lite JSON，因此只作为二期研究候选。

表格和电子表格：

- **ExcelJS 4.4.0**：MIT。适合服务端生成 `.xlsx`，支持 worksheet、style、image、公式、数据验证等；比直接让模型输出 CSV 更适合业务交付。
- **xlsx 0.18.5**：Apache-2.0。生态广，但历史上安全和维护风险需要单独审查；首选 ExcelJS，xlsx 只作为读取兼容候选。

白板和手绘图：

- **@excalidraw/excalidraw 0.18.1**：MIT。适合架构草图、流程白板、手绘风图解；source JSON 可作为事实源，SVG/PNG 为派生产物。
- **tldraw 5.0.1**：商业/SDK 边界、license 和复杂度需单独评审。适合后续完整白板编辑体验，不进入 M3/M4 默认实现。

公式：

- **KaTeX 0.16.47**：MIT。支持 server-side render to HTML，速度快，适合 TeX -> HTML/SVG/PNG 的轻量公式 artifact。
- **MathJax 4.1.2**：Apache-2.0。输入/输出更完整，适合 TeX/MathML/AsciiMath 和 SVG 输出；服务端复杂度高于 KaTeX，作为兼容增强。

报告：

- **ReportSpec 自研**：首发不要引入完整 CMS。报告应基于 Markdown/HTML section + figure/table references，输出 HTML/PDF/DOCX。
- PDF/DOCX 具体引擎需另开技术 PoC；原则是复用已有 asset、chart、table、diagram，而不是让模型再次复制内容。

结论：

- 扩展能力的优先级是 `ChartSpec` 和 `TableSpec`，因为它们能反哺 PPT/报告，提升内容可信度。
- `WhiteboardSpec`、`EquationSpec`、`ReportSpec` 进入 M4/M5，不抢 PPT/Mermaid/Mindmap 主线资源。
- 所有扩展都必须遵守统一平台合同：`Spec -> Run -> Worker -> Asset -> Canvas Preview -> Download`。
- 不能把扩展做成前端临时下载 blob；正式下载资产必须来自后端 asset service，带 owner、purpose、format、run id。
