## 6. 分阶段实施计划

### Phase 0：契约冻结

输出：

- 固化 `DeckSpec v1`。
- 固化 `DeckSpec` JSON Schema，并能被 `generate_ppt` tool schema 引用。
- 固化 `DiagramSpec v1`、`MindMapSpec v1` 和 Mermaid source 限制，并能被 `generate_diagram` tool schema 引用。
- 固化 `PresentationRun` 字段和 asset tag 关联规则。
- 固化 `presentation_runs` 数据表 schema、含 `resolving_images` 的状态机和同步/异步阈值。
- 固化 `DiagramRun` 字段和 asset tag 关联规则。
- 固化 `diagram_runs` 数据表 schema、状态机和同步/异步阈值。
- 固化 `RuntimeContext` 派生规则：auth user、KB runtime、session、trace、turn、tool call。
- 固化 presentation asset resolve 策略：`presentation_preview` / `presentation_download` / `presentation_audit`。
- 固化 diagram asset resolve 策略：`diagram_preview` / `diagram_download` / `diagram_audit`。
- 固化 Node worker tempfile 协议：stdin request、stdout JSON report、stderr logs、output path。
- 固化 diagram worker tempfile 协议：stdin request、stdout JSON report、stderr logs、source/svg/png/html output paths。
- 固化 worker 并发上限、非 0 exit/stdin 失败/stdout 非 JSON 的错误分类。
- 固化 asset 上传幂等合同：pending/committed、重试复用、GC 清理。
- 固化 editable PPTX XML 验收规则。
- 固化 `presentation/templates` registry 结构：
  - Swiss 主题 token 来自 `guizang-ppt-skill/references/themes-swiss.md`。
  - Swiss layout 白名单和图片槽位来自 `references/swiss-layout-lock.md`。
  - HTML deck 约定采用 `html-anything/src/lib/deck.ts` 的 `<section class="slide">` / `<aside class="notes">` 结构。
  - 每个 public layout 都有 PPTX inch 坐标、文本溢出策略、图片比例/裁切策略。
- 固化配额、保留期、GC dry-run 和 worker 依赖供应链策略。
- 固化 STRIDE/OWASP 威胁模型和发布阻断项。
- 准备 3 份 golden fixtures：
  - `swiss-basic-4.json`
  - `swiss-basic-8.json`
  - `editorial-basic-6.json`
- 准备 4 份 diagram golden fixtures：
  - `mermaid-flowchart-basic.mmd`
  - `mermaid-sequence-basic.mmd`
  - `mindmap-basic.mm.json`
  - `mindmap-basic.md`
- 明确首发只保证 Swiss 8 个版式可编辑输出。
- 明确首发不做流式预览、不公开 visual PPTX、不允许任意远程 HTTPS 图片。
- 明确 Mermaid 进入 M2 首发可用，Mindmap 进入 M3 或产品确认后前移到 M2。

验收：

- fixtures 可通过 JSON schema 校验。
- 每个 slide 指定 layout、主题、标题、正文、图片槽位。
- 契约文档覆盖 owner/session、asset purpose、worker protocol、PPTX editable 验收和错误分类。
- 契约文档覆盖 `presentation_runs` store、历史下载授权、异步 pending 状态和人工 PPTX 质量验收。
- 契约文档覆盖 `diagram_runs` store、历史下载授权、异步 pending 状态、SVG sanitize 和 MindMapSpec 限制。
- 契约文档覆盖配额、保留期、GC、供应链、license/NOTICE 和安全发布阻断项。
- 契约文档覆盖 schema 单一事实源、HTML deck version、历史消息恢复和模型重试上限。
- 未完成 Phase 0 不允许进入 UI 实现。

人工验收 fixtures：

- `swiss-basic-4.json`：M1 薄片 fixture，覆盖 Cover、Six Cells、Duo Compare、Image Hero。
- `swiss-basic-8.json`：M2 首发 fixture，覆盖首发 8 个 Swiss layout。
- `editorial-basic-6.json`：二期候选 fixture，只用于验证 registry 扩展性和避免后续重构；不代表首发公开支持 Editorial。

人工验收 checklist：

- 在 PowerPoint、Keynote、WPS 三者中至少一种打开成功。
- 每页标题可直接编辑。
- 正文/卡片文本可直接编辑。
- 图片可选中并替换。
- 色块、线条、卡片背景可选中。
- notes 能在演讲者备注中看到；若首发不支持 notes，必须有 warning。
- 中文不乱码、不严重溢出、不遮挡。
- 导出文件名和扩展名正确。
- Mermaid fixture 在 Canvas 中非空渲染，SVG 下载后不含 `<script>`、`foreignObject>`、`on*`、`javascript:`。

### Phase 1：后端 DeckSpec 与校验

修改/新增：

- `internal/presentation/spec/types.go`
- `internal/presentation/spec/outline.go`
- `internal/presentation/spec/planner.go`
- `internal/presentation/spec/validate.go`
- `internal/presentation/spec/schema.go`
- `internal/presentation/spec/validate_test.go`

校验规则：

- `title` 非空，长度限制。
- `language` 只能是 `zh-CN` / `en-US`。
- `style` 首发只能是 `swiss`；`editorial` 只允许在二期候选 fixture 和 registry PoC 中出现，不能进入公开 `generate_ppt` schema。
- Swiss theme 只能是 `ikb` / `lemon` / `lemon_green` / `safety_orange`。
- Swiss layout 首发只允许白名单：`S01,S03,S04,S08,S11,S15,S19,S22`。
- 每页文本长度按 layout 限制，防止溢出。
- 图片槽位必须匹配 layout。
- 复刻 `validate-swiss-deck.mjs` 的关键约束到 spec 层：
  - 不允许未登记 layout。
  - `S15/S16/S22` 图片槽位必须声明比例。
  - SVG/diagram 文本必须进入结构化 label 字段，不允许 renderer 生成 SVG `<text>`。
  - 正文页标题默认左对齐，只有 statement/cover/closing 类 layout 可居中或巨字排版。

测试：

- `go test ./internal/presentation/spec -v`
- 覆盖非法 layout、缺标题、过长标题、错误图片槽位、额外字段、非法本地图片路径。
- 断言 JSON Schema 的 layout enum 与 validator registry 一致。
- 断言 JSON Schema 由 `layouts.json` + config 动态生成，不允许 `generate_ppt.go` 手写第二份 schema。
- 对同一组 valid/invalid fixtures 同时运行 JSON Schema 和 Go validator，断言失败 JSON pointer 一致。
- 断言 validation error 带 JSON pointer。
- 断言 `OutlineSpec` 到 `DeckSpec` 的接口类型不依赖具体 renderer/exporter。
- 断言每个公开 layout 的 `pptx_slots` 坐标在 13.333 × 7.5 画布内。
- 断言每个 text slot 都有 `overflow_policy`，且只使用 `reject/split_slide`。

### Phase 2：HTML renderer

新增：

- `internal/presentation/renderhtml/renderer.go`
- `internal/presentation/renderhtml/swiss.go`
- `internal/presentation/renderhtml/templates/swiss.html`

实现策略：

- 不直接复制完整 `template-swiss.html` 的外部依赖。
- 提取其视觉 token、基础 CSS、版式骨架；首发不提取翻页 JS。
- 所有 slide 通过 Go template 或 strings.Builder 受控生成。
- 输出 HTML 结构统一为：
  - `<meta name="hive-deck-version" content="1">`
  - `<section class="slide" data-slide-id="1" data-layout="S01">`
  - 可选 `<aside class="notes">...</aside>`
  - 不输出外链 script，不输出用户 raw HTML。
- 渲染后运行轻量 HTML 自检：
  - slide 数一致
  - 每页有 `data-layout`
  - 每页有 `data-slide-id`
  - head 有 `hive-deck-version=1`
  - 无外部 script
  - 无未转义用户 HTML

测试：

- `go test ./internal/presentation/renderhtml -v`
- fixture 渲染后包含正确 slide 数、主题变量、layout 标识。
- fixture 渲染后包含 `hive-deck-version=1`，并可被前端 `parseDeck` 同等规则解析。

### Phase 3：PPTX exporter worker

新增：

- `tools/presentation-exporter/package.json`
- `tools/presentation-exporter/package-lock.json`
- `tools/presentation-exporter/src/exporter.mjs`
- `tools/presentation-exporter/src/layouts/swiss.mjs`
- `tools/presentation-exporter/src/schema.mjs`
- `tools/presentation-exporter/test/exporter.test.mjs`

实现策略：

- 使用 PptxGenJS。
- `pptx.layout = 'LAYOUT_WIDE'` 或自定义 13.333 × 7.5 inch。
- 每个 Swiss layout 映射为 PPTX 原生对象：
  - 标题：`slide.addText`
  - 卡片/色块：`slide.addShape`
  - 图片：`slide.addImage`
  - 分割线：`slide.addShape(line)`
- 保留主题 token 到 PPTX 色值。
- 中文字体优先 `PingFang SC` / `Microsoft YaHei` fallback，英文 `Inter`/`Arial`。
- 实现 `--health`：输出 `{ "ok": true, "protocol_version": 1, "pptxgenjs_version": "4.0.1" }`。
- worker stdout 只输出 JSON report，PPTX 写入 Go 指定 output path。

测试：

- `npm test` in `tools/presentation-exporter`
- 用 Node 测试生成 `.pptx`。
- 解压 PPTX 检查：
  - `[Content_Types].xml`
  - `ppt/presentation.xml`
  - `ppt/slides/slide1.xml`
  - 每页至少包含标题文本节点。
  - editable 模式不得只有 full-slide 图片。
- 对每个公开 layout 读取 `layouts.json` 的 `pptx_slots`，断言输出对象边界不越界。
- 对超长文本 fixture 断言 validator 拒绝，不能由 exporter 静默缩小字体。
- 中文 fixture 验证 PPTX 中字体声明包含中文 fallback，PowerPoint/Keynote/WPS 至少一种打开不乱码。
- notes fixture 验证 speaker notes 写入成功；若某环境不支持，必须输出 warning 且测试覆盖 warning。

### Phase 4：Go 侧 PPTX 封装与 asset 上传

新增：

- `internal/presentation/runtime/context.go`
- `internal/presentation/pptx/exporter.go`
- `internal/presentation/pptx/node_worker.go`
- `internal/presentation/assets/upload.go`
- `internal/presentation/service.go`
- `internal/store/presentation_runs.go`

实现策略：

- `Service.Generate(ctx, request) (PresentationResult, error)` 负责编排 validation、runtime、image resolve、HTML render、PPTX export、asset upload。
- `PresentationRunStore` 记录 run 状态、错误、耗时、asset URI 和统计字段。
- `Exporter.Export(ctx, request) (ExportResult, error)` 使用固定 worker 协议。
- 默认 worker path 从 config 读取，开发默认 `tools/presentation-exporter/src/exporter.mjs`。
- Go 控制超时、stdout/stderr、output path、最大输出。
- 图片 resolve 在 `resolving_images` 阶段集中完成，HTML/PPTX 共用 `resolved_assets`。
- 上传 DeckSpec JSON、HTML、PPTX 到 `asset.AssetService`，三者共享 `presentation_run_id`，并按 pending/committed 合同处理重试。
- asset upload owner 来自 `RuntimeContext`，不是模型输入。
- 同步生成超过 `sync_timeout_seconds` 时返回 `status:"running"`，由前端轮询 run。

测试：

- `go test ./internal/presentation/... -v`
- 用 fake exporter 测 asset 上传路径。
- 用 fake store 测状态流转：created -> running(stage=validating) -> running(stage=resolving_images) -> running(stage=rendering_html) -> running(stage=exporting_pptx) -> running(stage=uploading_assets) -> succeeded(stage='')。
- 用 fake store 测失败流转和错误分类。
- 用 fake store 测 worker 原子领取、租约续期、租约过期恢复和并发 worker 不重复执行。
- 用 fake quota 测并发 running 上限和每日生成上限。
- 用 fake image resolver 测必填图片失败、可选图片降级、图片只解析一次。
- 用 fake asset service 测上传 2/3 后失败、重试复用 pending asset、GC dry-run 列出 pending asset。
- 用 fake worker 测非 0 exit、stdout 为空、stdin 写入失败、worker protocol version 不兼容、`exporter_busy`。
- 用真实 worker 的 integration test 可加 build tag，避免普通单测依赖 Node。
- 测 worker timeout、stdout 非 JSON、output 超限、PPTX XML 验收失败。
- 测无 auth/user fallback 场景的错误分类。

### Phase 8：端到端薄片与首发验收

验收用例：

1. 用户请求“做一份 6 页瑞士风 PPT”。
2. 模型调用 `generate_ppt`。
3. 工具生成 HTML 和 PPTX asset。

实现策略：

- 用 fixture deck 截 2 页 PNG，断言 blob 非空且尺寸接近 16:9。
