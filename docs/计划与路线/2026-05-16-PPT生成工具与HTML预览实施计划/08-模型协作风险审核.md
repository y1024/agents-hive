> 上级索引：[2026-05-16-PPT生成工具与HTML预览实施计划.md](../2026-05-16-PPT生成工具与HTML预览实施计划.md)

## 7. 提示词与模型协作

系统 prompt 需要补充工具使用纪律：

- 当用户请求生成 PPT/演示文稿/slide deck 时，优先调用 `generate_ppt`。
- 当用户请求生成流程图、时序图、架构图、Mermaid 图、脑图、思维导图或可下载图表时，优先调用 `generate_diagram`。
- 用户明确要求“Mermaid 源码”且不需要预览/下载时，可以直接输出 fenced Mermaid；用户要求“生成图 / 下载图 / 右侧预览”时必须调用 `generate_diagram`。
- 用户明确要求“脑图/思维导图”时，优先生成 `diagram_type:"mindmap"`；如果 mindmap worker 未启用，应返回结构化不可用错误或改问用户是否接受 Mermaid flowchart 替代，不能静默输出普通 Markdown。
- 不要把完整 HTML 直接塞进普通回答，除非用户明确只要 HTML。
- 先产出结构化 DeckSpec，再调用工具。
- 先产出结构化 DiagramSpec，再调用 `generate_diagram`。
- 如果用户要求 PPTX 可编辑，必须使用 `mode:"editable"`。
- 如果图片缺失，可先生成无图 deck，但应在 warnings 里说明图片槽位为空。

可给模型的 DeckSpec 生成约束：

- 中文标题短句优先。
- 每页最多一个主观点。
- Swiss 风格使用锁定版式，不临时发明结构。
- 不编造 KPI 数据；无真实数据时用概念卡片版式。

可给模型的 DiagramSpec 生成约束：

- Mermaid 语法优先选择 `flowchart`、`sequenceDiagram`、`stateDiagram-v2`、`classDiagram` 这类稳定语法。
- Mermaid 节点 label 保持短句，不塞长段落。
- 不使用 Mermaid `click` action、HTML label、外部图片或外部链接。
- MindMapSpec 节点应是名词短语或动作短语，单节点不要超过 120 字符。
- MindMapSpec 深度优先控制在 4 层以内；复杂内容拆成多个图或改为 PPT。
- 需要下载 SVG/PNG/HTML 时通过 `outputs` 明确请求，不让前端临时导出作为唯一产物。

## 8. 风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| HTML 预览和 PPTX 视觉不完全一致 | 用户看到的和下载的不同 | DeckSpec 同源；首发限制版式；为每个 layout 建 screenshot/golden 检查 |
| PPTX 可编辑性和视觉还原冲突 | 原生对象难以达到 WebGL/复杂 CSS 效果 | 明确 editable 为主；复杂背景降级为静态形状或可选图片背景 |
| Node worker 增加部署复杂度 | server binary 不再完全自包含 | config 开关、自检、Docker 安装 Node、worker 单独版本化 |
| 生成 HTML scripts 安全面 | iframe 执行脚本风险 | 首发无脚本，React 控制翻页；后续只允许 trusted generated deck 使用 `allow-scripts`，无 same-origin |
| 大文件/图片导致卡顿 | Canvas 或下载慢 | 限制图片数量和大小；asset 转存；PPTX 最大 50MB |
| 模型输出 DeckSpec 不稳定 | 工具失败率高 | 强 schema、错误可恢复、给模型 examples、必要时工具支持从简化 outline 自动排版 |
| 版式范围过大 | 工期膨胀 | 首发 Swiss 8 个 layout，其余后续按需求增加 |
| asset 权限过宽 | 用户可能下载到不该暴露的 presentation asset | `generated_presentation` 专门 resolver，校验 owner、purpose、format、session |
| 图片 URL SSRF | 服务端可被诱导访问内网 | Go image resolver 做 allowlist、DNS/IP 私网拦截、大小限制、MIME sniff |
| 前端把 presentation 当普通 HTML | 下载退化为 `.html` 或 `.md` blob | `ArtifactType='presentation'` + metadata + `useAsset` resolve 下载 |
| 只存 HTML/PPTX 不存 DeckSpec | 无法重试、编辑、审计 | DeckSpec JSON asset 或 `presentation_runs` record 强制保存 |
| 文本溢出 | PPTX 可打开但不可读 | layout registry 定义最大长度，validator 拒绝或自动拆页，不在 exporter 临时压缩 |
| license 漏归属 | 合规风险 | `template.json` 记录来源，新增 NOTICE/文档归属说明 |
| 流式预览期望错位 | 用户以为边生成边预览 | 首发明确非流式，生成完成后立即打开 Canvas；流式预览进入 M3 |
| visual PPTX 混淆用户 | 用户下载不可编辑文件后以为功能失败 | 首发不公开 visual 下载，只作为内部 QA/fallback |
| 缩略图 iframe 过多 | Canvas 卡顿、内存上涨 | 首发缩略图用页码/标题；最多只渲染当前页附近 iframe |
| 文档不可追踪 | 计划无法提交和协作 | 已在 `.gitignore` 为本计划文件添加例外；后续新增架构/运维文档也必须添加例外或迁移到可跟踪路径 |
| Mermaid/脑图被当成普通代码块 | 用户无法在 Canvas 管理和下载正式资产 | `generate_diagram` + `DiagramArtifactMeta` + `diagram_runs`，普通 fenced Mermaid 只做兼容 |
| SVG XSS | Canvas 预览或下载 SVG 执行恶意内容 | strict Mermaid、服务端/前端双重 sanitize、asset purpose 隔离 |
| Mindmap 节点爆炸 | worker/Canvas 卡死 | `max_nodes/max_depth/max_source_kb`、worker timeout、前端虚拟化或折叠 |
| Mermaid/Mindmap worker 供应链 | Playwright/Mermaid/Markmap 依赖风险 | 独立 worker、package-lock、npm audit、health check、固定 worker path |
| PNG/HTML 导出过早开放 | Playwright 稳定性和安全面扩大 | M2 只开 source/SVG，PNG/HTML behind config 进入 M3 |
| Artifact 工具超市化 | 能力散、维护成本高、用户不知道选什么 | 只接能闭环到 Canvas/下载/PPT/报告复用的 artifact；每类必须有 Spec/Run/Asset |
| 模型编造数据图表 | 用户把虚构数据当真实业务结论 | `data_provenance` 强制标记，模型生成数据 UI 显示来源，真实数据优先来自上传/KB/tool |
| Excel/CSV formula injection | 下载表格被打开时触发公式执行 | 默认禁公式，`= + - @` 开头文本按纯文本转义，M4 再做公式白名单 |
| ECharts option 过宽 | 模型生成任意 JS-like 配置或外链资源 | M3 用受限 ChartSpec + Vega-Lite canonical renderer；ECharts M4 只开放白名单 chart option |
| 白板 source 膨胀 | Excalidraw/tldraw JSON 过大导致 Canvas 卡顿 | 元素数、图片数、总 bytes 限制，图片先 asset 化，超限拒绝 |
| 公式 macro 滥用 | TeX/MathJax macro 造成安全或渲染 DoS | macro allowlist、source bytes、render timeout、SVG sanitize |
| Report 复制失去 provenance | PDF/DOCX 无法追踪图表/表格来源 | ReportSpec 引用 artifact URI，导出 metadata 保留 source run ids |

## 8.1 Claude Code 工程审核处理结果

审核报告归档：`10-Claude工程审核原始报告.md`。

已采纳并补入主计划：

- P1 `Asset 上传与 Run 状态更新的事务边界`：补 `presentation_asset_state=pending|committed`、重试复用、最终原子更新 run、GC 清理 pending asset。
- P1 `图片 resolve 阶段不明确`：新增 `resolving_images` stage，HTML/PPTX 共用 resolved assets。
- P1 `Worker crash 处理测试缺失`：补非 0 exit、stdout 为空/非 JSON、stdin 写入失败、protocol 不兼容测试。
- P1 `前端 HTML parse 失败静默空白`：补 `hive-deck-version=1`、`unsupported_deck_html_version`、`empty_deck_html`、`invalid_deck_html`。
- P2 `Node worker 无全局并发控制`：补 `max_concurrent_workers` 和 `exporter_busy`。
- P2 `Deck HTML 结构隐式合同`：补 HTML deck version 和前后端解析测试。
- P2 `JSON Schema 与 Go 类型同步机制`：补 `layouts.json + config` 为 schema/validator 唯一事实源。
- P2 `同步/异步恢复路径`：补所有 presentation artifact 保存 `runId`，历史消息和 asset resolve 失败走 run 状态恢复。
- P2 `限制值散落`：补 config 注入 schema/service/worker，不手写多份。
- P2 `recoverable 语义`：补 `recoverable_by`、模型自动重试 2 次、同 session 连续失败上限。
- P2 `Proxy URL 签名验证`、`parseDeck 单测`、`中文字体 fallback`、`图片失败降级`、`移动端 DeckRenderer`：补入对应 Phase 测试。

保留为实现阶段取舍但边界已写清：

- ToolResultCard 抽取仍作为首发前置任务；如果实施分支冲突严重，可先单独提交抽取以降低冲突。
- Per-request spawn Node 首发保留，M3 再评估长驻 worker + JSON-RPC。
- `mode` 数据库层预留 `visual`，但 Go/tool schema/UI 首发仍只允许 `editable`。

## 8.2 脑图 / Mermaid 追加审核处理结果

用户追加指出：本计划不能只覆盖 PPT，还必须覆盖脑图、Mermaid 等可视化 artifact 的生成、预览和下载。该意见正确，且不是边缘需求。已按强范围补入主计划：

- 将目标从“PPT 生成工具”升级为 `VisualizationArtifact` 平台：PPT 走 `DeckSpec` / `generate_ppt`，图表和脑图走 `DiagramSpec` / `generate_diagram`。
- 增加 `diagram_runs`，作为 Mermaid/Mindmap 的审计、重试、异步恢复和下载授权单位。
- 增加 `generate_diagram` 输入/输出 schema，覆盖 Mermaid source、MindMapSpec JSON、source/SVG/PNG/HTML asset URI、结构化错误和模型修复规则。
- 增加 `tools/diagram-exporter` worker 协议、部署配置、health check、版本锁定和 `npm audit` 发布要求。
- 增加 `diagram_preview` / `diagram_download` / `diagram_audit` asset purpose，区分 SVG preview inline 和 source/SVG/PNG/HTML attachment 下载。
- 增加 `DiagramRenderer`、`MermaidRenderer`、`MindMapRenderer`、`useDiagramRun` 和历史消息恢复路径。
- 将 Mermaid Canvas 预览和 `.mmd/.svg` 下载锁定为 M2 必须项。
- 将 Mindmap Canvas 预览和 `.mm.json/.md/.svg` 下载锁定为 M3 必须项；若产品优先级高于 PPT 首发，可前移到 M2。
- 增加 SVG XSS、Mindmap 节点爆炸、危险链接、Mermaid/Markmap/Playwright supply chain 的威胁模型和发布阻断项。
- 增加 Mermaid/Mindmap fixtures、Go/TS 单测、asset resolver 负测和 Playwright E2E 验收。

审核结论：如果后续实施只做 PPT，而没有 Mermaid M2 和 Mindmap M3 的 run/schema/asset/Canvas 闭环，则不满足本文 12 分交付契约。

## 8.3 扩展 Artifact 追加审核处理结果

用户追问“还有没有其他可以集成的”，从顶级产品经理视角，正确答案不是继续堆开源库，而是把 Hive 定义为 `AI Artifact Studio`。已补入主计划：

- 增加 `ChartArtifact`：`generate_chart`、`ChartSpec`、`chart_runs`、Vega-Lite source、SVG/PNG 下载，M3 优先。
- 增加 `TableArtifact`：`generate_table`、`TableSpec`、`table_runs`、ExcelJS `.xlsx`、`.csv/.json` 下载，M3 优先。
- 增加 `WhiteboardArtifact`：`generate_whiteboard`、`WhiteboardSpec`、Excalidraw source、SVG/PNG 下载，M4。
- 增加 `EquationArtifact`：`generate_equation`、`EquationSpec`、TeX/SVG/PNG 下载，KaTeX 默认、MathJax 增强，M4。
- 增加 `ReportArtifact`：`generate_report`、`ReportSpec`、PDF/DOCX/HTML 下载，M5。
- 明确所有扩展必须复用 `Spec -> Run -> Worker -> Asset -> Canvas Preview -> Download` 合同，禁止前端临时 blob 冒充正式下载。
- 明确数据 provenance：真实数据来自上传、KB、tool result 或人工输入；模型生成数据必须显式标记。
- 明确暂缓 PlantUML、Graphviz、draw.io、Figma、任意 React app artifact，避免安全和范围失控。

审核结论：PPT/Mermaid/Mindmap 是第一层办公表达能力；Chart/Table 是第二层数据可信能力；Whiteboard/Equation/Report 是第三层场景扩展。实施顺序不能反过来，否则会先做出花哨但不可信的交付物。

## 9. 不做事项

首发不做：

- 任意 HTML 到可编辑 PPTX 的通用转换。
- WebGL 动效完整导出到 PPTX。
- 多人协作编辑。
- PowerPoint 主题母版的完整企业模板导入。
- M2 首发不做复杂图表自动生成；Chart/Table 进入 M3。
- M3 不做完整白板协作、复杂公式宏、完整报告排版系统；Whiteboard/Equation 进入 M4，Report 进入 M5。
- 所有 `guizang-ppt-skill` 22 个 Swiss 版式的 PPTX 映射。
- PlantUML、Graphviz、draw.io、Figma、任意 React app artifact 默认不进入 M3/M4。
