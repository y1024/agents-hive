# Hive 平台多业务承载演进 v1

## 背景与触发

用户问"Hive 能不能承接 AI 客服能力，不改架构、不大量重构"。重新框定后实质问题：
**Hive 当前架构能否承载持续多元业务能力叠加（客服、销售、HR、运维等）。**

业务数量预期：不确定，但可能走多业务路线（≥3 个业务）。

## 平台扩展点（已就位 / 复用，不动）

1. **Skills Overlay Registry**（`internal/skills/overlay_registry.go`）—— 4 层 + tenant 隔离 + pg_notify 热更新
2. **Channel Plugin 接口**（`internal/channel/plugin.go`）—— 飞书/钉钉/企微/微信现成
3. **AIRouter**（`internal/airouter/`）—— LLM/embedding/image/stt/tts/video 多模态路由
4. **Spec-driven Cognition**（`internal/specdriven/` + `openspec/`）—— 新能力走规范化流程
5. **Master EventBus**（`internal/master/event_bus.go`）—— 业务事件总线已存在
6. **PromptLoader 三层**（`system/business` 段是预留 hook）
7. **MCP 工具协议** + 自定义工具目录
8. **Subagent 工厂**（`internal/subagent/factory.go`）—— 新 Agent 类型可注册

## Scope（基线 + 已 cherry-pick 的扩展）

### 平台定位（关键）

**Hive = 能力平台 + API/Webhook，UI 由集成方实现。**
v1 不做任何客服坐席视图、KB 管理 UI、业务域指示器等前端工作。

### 基线（接客服 1 个业务，零架构改动）

- `customer-service` Skill Pack（话术 + 业务路由 trigger_keywords）
- `kb_search` tool（基于 pgvec，作为 KB 模块的客户端）
- `escalate_to_human` tool（推飞书值班群）

- **Webhook 出站**：集成方注册 callback URL，escalate 触发/状态变更/AI 回复时推送事件
- **OpenAPI spec**：`openapi/cs-v1.yaml`，集成方自生成 SDK

### Cherry-pick：现在就做（瓶颈 3 — KB 与 Memory 解耦）

**Why（现在做）：** 多业务路线必经；零负担（不动 memory）；为后续业务隔离打底；多业务接入时不需要回头重构 memory 表。

- 新增 `internal/kb/`：namespace（业务域）+ version + permission + ingest 管线
- 复用 pgvec 但**新表**（`kb_documents` / `kb_chunks` / `kb_namespaces`）
- 不动 `internal/memory/`（agent 记忆语义保留）
- KB 与 Memory 通过 tool 接口隔离：`kb_search` vs `memory_search`

### Cherry-pick 但延后：瓶颈 1（Prompt 膨胀）+ 瓶颈 2（关键词路由冲突）

**触发条件（写入文档）：**

## Review 决策记录

### Section 2（错误与 Rescue）
- **KB Ingest 部分失败**：事务全部回滚，任何 chunk embedding 失败 → 整份文档 ingest 失败，返回 422 + 明确失败 chunk 索引/原因。需要：`pg.BeginTx` 包裹 ingest 全流程
- **LLM 幻觉引用 KB**：KB 检索结果注入时每个 chunk 带 `[doc_id:xxx]` 标识，prompt 强约束 LLM 引用时必须输出 `[doc_id:xxx]` 标注，master 层 post-processor 验证响应中所有 doc_id 都来自本次检索结果，不匹配则标"⚠️ 可能引用错误"。需要：`internal/cs/rag_validator.go`

### Section 1（架构）
- **Escalate 失败处理**：飞书 SDK 重试 3 次 + 失败后向 user 明示"已报警人工但推送延迟，你可以继续与 AI 对话" + 后台占台重试队列（指数退避，最长 30min）。新增：`internal/cs/escalation_queue.go` + `pg` 表 `cs_escalation_retries`
- **AI 处理中超时**：60s 软超时（向 user 推送"还在思考"）+ 180s 硬超时（强制转人工）。独立于现有 ReAct 全局 5min 超时

### Section 3（安全）
- **文档解析沙箱**：复用 `docker/sandbox/Dockerfile` 现成镜像，`internal/kb/ingest.go` 调 `internal/sandbox` 接口，30s 超时，OOM/越权 → 整份 ingest 失败 422。理由：unidoc/pdfcpu CVE 频发必须强隔离，零新技术栈
- **PII 脱敏**：v1 不做统一模块（用户明确不收集客户身份证等 PII）。escalate reason/日志中偶发手机号做最小正则过滤即可，列 P3 TODO
- **跨租户 KB 访问**：`kb_search` tool 必须强制注入 `tenant_id` from session context，DB 层 `WHERE tenant_id=$1` 强约束，单测覆盖跨租户场景
- **Prompt Injection（KB 投毒）**：v1 用"KB 内容显式包裹 + LLM 引用必须带 doc_id + post-validator"三层防御（已在 Section 2 决策），不做 KB 内容入库前的 LLM 安全扫描（成本太高）

### Section 4（数据流与边界）
- **escalate 解耦**：目的地不硬编码飞书；集成方走 webhook 自处理，飞书推送降级为默认 fallback
- **session 单写入者**：IM Channel 与 API 双路径不能同时写一个 session，后到者 409
- **业务域不可变**：`session.metadata.business_domain` 一旦写入冻结，跨域必须新开 session（写入 OpenAPI doc）
- **webhook 至少送达一次**：失败入 `webhook_dlq` 表，集成方拉取；事件带 `event_id` UUID，集成方负责幂等
- **关键不变量**：tenant 隔离 / doc_id 引用闭环 / escalate 单调 / 业务域不可变 / webhook 至少送达一次

### Section 5（代码质量）

## Outside Voice 模拟（独立审查）

以独立架构师视角对本 plan 提 5 个尖锐问题（自答）：

1. **Q: KB 与 Memory 真的解耦了吗？还是只是新建表伪解耦？**
   A: 真的。语义维度：memory 存 agent 自我认知（4 类型），KB 存企业可检索知识。访问维度：tool 入口分离（`memory_search` vs `kb_search`）。表维度：完全独立，无外键。如果未来需要"memory 引用 KB"，是显式 cross-reference 不是混表。

2. **Q: cs state machine 用 JSON metadata 是否会演变成一坨字符串汤？**
   A: v1 状态机最多 5 个 state、表驱动、纯函数。如果 Phase 2 状态超过 10 或并发状态出现，再考虑 ENUM 升级。门槛明确，不是"以后再说"。

3. **Q: webhook 是不是给 v1 加了不必要的复杂度？为什么不让集成方拉取？**
   A: 因为 escalate 是即时性强的事件，拉取模型延迟不可接受。webhook 是工业标准，DLQ 兜底。如果集成方真的想拉，DLQ API 已经满足。

4. **Q: 飞书推送降级为 fallback，那是不是把"飞书 SDK 唯一"原则给削弱了？**
   A: 没有。原则是 Hive **内部**调飞书必须走 SDK，不是禁止集成方自己处理。webhook 是把"通知谁"决策交给集成方，与"内部调用合规"是正交的。

5. **Q: 如果 6 个月只接了客服一个业务，这套 KB + cs 是不是过度工程？**
   A: 不是。即使只做客服，KB 模块独立有价值（企业知识库 SaaS 形态可独立售卖），cs 模块的 escalate/state 是客服底线刚需。瓶颈 1/2 已经延后，没付未来代价。

**独立审查结论：通过。** 三处建议：(a) OpenAPI 例子要全；(b) rag_validator 违规率告警阈值 1% 偏严，建议 v1 设 5% 观察 1 个月再调；(c) escalate 队列 30min 上限是否够用需看真实集成方 SLA，建议参数化。
