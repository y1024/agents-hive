### A1【P0 / 9 分】RuntimeExec + IntentManageTool 矩阵与算法自相矛盾

**证据：**

`internal/router/decision.go:200-201`（当前实现）：
```go
if profile.Risk == RiskRuntimeExec && intent.Kind != IntentManageTool {
    return "runtime execution not required by intent"
}
```
当前对 `IntentManageTool` 放行 RuntimeExec 工具。

计划 §1 矩阵：
> runtime exec | `bash`, `shell`, `run_command` | **只有明确 runtime/manage intent 才进入候选** | SafeExecutor allow/ask/deny

计划 §6 Task 2 step 1 `EvaluateToolPolicy`：
```go
if profile.OpenWorld || profile.Destructive || profile.Risk == RiskRuntimeExec || profile.Risk == RiskDestructive {
    return policyDecision(ToolPolicyDeny, ToolRouteBlockedDangerous, false, false, ...)
}
```
**无视 intent，一刀切 deny。**

**蓝军 mutation 表：**

| 输入 | 矩阵期望 | 当前 blockReason 行为 | 计划新行为 |
|------|---------|----------------------|-----------|
| `bash` + `IntentManageTool` | 进入候选 | 不 block（穿透到 isCallable） | **被 dangerous gate 拒在 blockReason** ← 回归 |
| `bash` + `IntentRead` | 不进入候选 | block "runtime execution not required by intent" | block "dangerous_or_open_world_tool" |

**后果：**
- 矩阵承诺的"runtime/manage intent 可进入候选 → SafeExecutor 决定"链路被砍。
- `internal/agentquality/route_decision_test.go:369` 的 `runtimeExecProfile` + IntentManageTool 测试会暴露回归。

**推荐修复（必须在 Task 2 实施前回灌）：**

```go
// 在 EvaluateToolPolicy 中：
if profile.OpenWorld || profile.Destructive || profile.Risk == RiskDestructive {
    return policyDecision(ToolPolicyDeny, ToolRouteBlockedDangerous, ...)
}
if profile.Risk == RiskRuntimeExec {
    if ctx.Intent.Kind == IntentManageTool {
        return policyDecision(ToolPolicyAsk, ToolRouteRequiresMatchingIntent, true, true, "runtime_exec_under_manage_intent", profile)
    }
    return policyDecision(ToolPolicyDeny, ToolRouteBlockedDangerous, ...)
}
```

**Task 1 测试矩阵补 2 用例：**
- `runtime_exec_with_manage_intent_allows_under_safe_executor`
- `runtime_exec_with_read_intent_blocked`

### A2【P0 / 9 分】feishu_api 外发动作失去 HITL 兜底 → 静默外发

**证据：**

`internal/router/capability_registry.go:133-144`（`structuredDangerousActions`）：
```go
"feishu_api": {
    "create_approval", "create_bitable_record", "update_bitable_record",
    "create_task", "complete_task", "write_sheet",
},
```
**注意：`send_message`/`send_image`/`send_file`/`upload_image`/`upload_file` 不在表里。**

`internal/router/capability_registry.go:172-177`（`mixedActionRules.ExternalSendActions`）：
```go
ExternalSendActions: []string{
    ..., "upload_image", "upload_file", "send_message", "send_image", "send_file",
},
```

这些"对外发送"动作目前依赖 `internal/master/action_guard.go:64-66` 的 `isExternalSendAction` 兜底返回 Ask。

计划 §10 Task 6 step 3 写：
> Keep explicit `isExternalSendAction` and `StructuredDangerousOperation` **only if tests prove they are needed for exact HITL behavior; otherwise prefer** `ToolActionProfile + EvaluateToolPolicy`.

**蓝军 mutation：删除 `isExternalSendAction`，回放 `feishu_api` + `{"action":"upload_file"}`：**

1. `InferToolProfile` → builtin rule → `Risk=RiskExternalWrite, SideEffect=true`
2. `toolArgumentsRequireApproval(input)`：
   - key="action", value="upload_file"
   - `toolActionLooksDangerous("upload_file")`：tokens=["upload","file"]
   - 关键字列表：`delete, drop, truncate, update, insert, write, create, send, publish, deploy, restart, exec, execute, shell`
   - **"upload" 不在列表** → 返回 **false** → 不触发 Ask
3. `EvaluateToolPolicy`：
   - `ToolActionProfile`：action 走 `ExternalSendActions` 分支 → Risk=RiskExternalWrite, SideEffect=true
   - dangerous gate：false（RiskExternalWrite 不在列表）
   - `IsMixedReadWriteTool` → true → `evaluateMixedToolPolicy`
   - `StructuredDangerousOperation` → **false**（注册表没 upload_file）
   - `MixedAllowedToolInputsForIntent(intent={}, "feishu_api")`：return nil
   - `MixedReadOnlyActions("feishu_api")`：非空 → **返回 `ToolPolicyAllow, mixed_read_actions_available`**
4. ActionGuard 拿到 Allow → **直接执行 feishu_api.upload_file 不弹窗确认**

**这是 IM 外发副作用绕过 HITL 的安全回归。**

**推荐修复（方案 A，Layer 1，最小改动）：** 把 `isExternalSendAction` 的语义沉到 `evaluateMixedToolPolicy`：

```go
func evaluateMixedToolPolicy(profile ToolProfile, ctx ToolPolicyContext) ToolPolicyDecision {
    if StructuredDangerousOperation(profile.Name, ctx.Input) { ... }
    // 新增：当前 action 若在 ExternalSendActions 中，强制 ask + requires_side_effect_intent
    if action := structuredAction(ctx.Input); action != "" &&
        containsActionString(ExternalSendActions(profile.Name), action) {
        callable := ctx.Intent.AllowsSideEffects && ctx.Intent.Kind == IntentExternalWrite
        decision := policyDecision(ToolPolicyAsk, ToolRouteRequiresSideEffectIntent, callable, true, "external_send_action", profile)
        decision.RequiresSideEffectIntent = !callable
        return decision
    }
    ...
}
```

**否决方案：**
- **方案 B**（把 ExternalSendActions 全部并入 `structuredDangerousActions`）— 会让 `IntentExternalWrite` 路径回归，需重构 mixedActionRule 含义，**成本高、风险高**。
- **方案 C**（保留 `isExternalSendAction` 在 ActionGuard）— **违背"统一"初衷**，是 shortcut。

→ **强制方案 A。**

### A3【P1 / 8 分】`ProfileRequiresApproval` 对 destructive/unknown 工具语义回归

**证据：**

当前实现（`internal/router/capability_registry.go:471`）：
```go
if structuredDangerousTools[...] { return true }
if IsMixedReadWriteTool(profile.Name) { return false }
return ProfileHasSideEffect(profile)
```
对 `RiskDestructive`：`ProfileHasSideEffect` 返回 true → 整体返回 **true**。

计划新实现：
```go
decision := EvaluateToolPolicy(profile, ToolPolicyContext{})
return decision.RequiresApproval
```
对 `RiskDestructive`：`EvaluateToolPolicy` 走 dangerous gate → `Action=Deny, RequiresApproval=false` → 整体返回 **false**。

**蓝军：** `internal/gateway/methods_mcp.go:136`、`internal/router/capability_gate_test.go:221/247` 都调 `ProfileRequiresApproval`。返回 false 的 destructive 工具，在 gateway 元数据里被标"不需要审批" → UI 不显示警告 → 用户被骗。

**Why：** "需要审批"和"会被拒"是两件事。destructive 工具 deny ≠ 不需要审批，它需要"高级别审批 / 越过 deny"。

**推荐修复：**

```go
func ProfileRequiresApproval(profile ToolProfile) bool {
    if structuredDangerousTools[strings.TrimSpace(strings.ToLower(profile.Name))] {
        return true
    }
    decision := EvaluateToolPolicy(profile, ToolPolicyContext{})
    return decision.RequiresApproval || decision.Action == ToolPolicyDeny
}
```

### A5【P1 / 7 分】算法依赖隐式不变量，无注释

**证据：**

`internal/router/capability_registry.go:483-513` `ToolActionProfile`：
```go
if profile.Name == "" || !IsMixedReadWriteTool(profile.Name) { return profile }
if StructuredDangerousOperation(profile.Name, input) { return profile }  // ← dangerous op 时保留原 profile
...
```

`memory.delete` 的链路：
1. `ToolActionProfile`：`StructuredDangerousOperation` true → 不动 profile（原本是 RiskLocalWrite 等，不是 RiskDestructive）
2. dangerous gate：profile.Risk ≠ RiskDestructive → 不拒
3. `IsMixedReadWriteTool` → true → `evaluateMixedToolPolicy`
4. `StructuredDangerousOperation` → true → Ask ✓

**算法碰巧对，因为 `ToolActionProfile` 对 dangerous op 故意保留原 profile。** 这是隐藏耦合，没有注释，没有不变量声明。任何人未来把 `ToolActionProfile` 改成"dangerous op 时上调到 Destructive"，整个矩阵会跳到 Deny。

**推荐：** 在 `EvaluateToolPolicy` 顶部加注释：

```go
// IMPORTANT: 算法依赖以下不变量
// 1. ToolActionProfile 对 mixed 工具的 dangerous 动作返回原 profile（不升级到 Destructive）
//    —— 这样混合工具不会被外层 dangerous gate 一刀拦截
// 2. 外层 dangerous gate 优先于 mixed 分支：保护非混合工具的 Destructive/Runtime 一定 Deny
// 3. ProfileHasSideEffect 对 RiskExternalWrite/RiskLocalWrite 返回 true
//    —— 这样 mixed 之外的写工具能被 side-effect 分支接住
//
// 改这三条任意一条，先把 tool_policy_test 跑一遍。
```

并在 `ToolActionProfile` 的 doc 也补这条不变量。

### B2【P1 / 7 分】`Decide` 顺序错位：arg-scan 早于 policy，掩盖 Deny

**证据：** 计划 §10 Task 6 step 3：

```go
if input.ToolDef != nil {
    profile := router.InferToolProfile(*input.ToolDef, router.ProfileHint{})
    if toolArgumentsRequireApproval(input.Arguments) {
        return actionGuardDecision(ActionGuardAsk, "argument_side_effect", ...)
    }
    policy := router.EvaluateToolPolicy(profile, ...)
    switch policy.Action { ... }
}
```

**蓝军：** `metamcp__delete_dashboard({"action":"delete"})`
- `toolArgumentsRequireApproval`：key="action", value="delete" → "delete" 在 keyword 列表 → 返回 true → **Ask**
- 但 profile 是 `RiskDestructive`，`EvaluateToolPolicy` 本应返回 **Deny**
- **早返回的 Ask 比晚返回的 Deny 更宽松 → 安全降级**

虽然这是既有行为（`action_guard.go:75-76`），但旧代码对 trusted MCP 单独走 if 块，结构上看得清。新代码把它泛化到所有工具，**问题面更大**。

**推荐修复：** 顺序对调 —— 先 policy，policy 是 Deny 直接 Deny；只有 Allow 时才用 arg-scan 升级到 Ask：

```go
if input.ToolDef != nil {
    profile := router.InferToolProfile(*input.ToolDef, router.ProfileHint{})
    policy := router.EvaluateToolPolicy(profile, router.ToolPolicyContext{Input: input.Arguments, ForAction: true})
    if policy.Action == router.ToolPolicyDeny {
        return actionGuardDecision(ActionGuardDeny, policy.Reason, "tool_policy", "")
    }
    if toolArgumentsRequireApproval(input.Arguments) {
        return actionGuardDecision(ActionGuardAsk, "argument_side_effect", "tool_policy", "")
    }
    switch policy.Action {
    case router.ToolPolicyAllow:
        return actionGuardDecision(ActionGuardAllow, policy.Reason, "tool_policy", "")
    case router.ToolPolicyAsk:
        return actionGuardDecision(ActionGuardAsk, policy.Reason, "tool_policy", "")
    }
}
```

### B5【P2 / 6 分】`evaluateMixedToolPolicy` fallback 把未知 action 当 side-effect

```go
return policyDecision(ToolPolicyAsk, ToolRouteRequiresSideEffectIntent, ctx.Intent.AllowsSideEffects, true, "mixed_side_effect_requires_intent", profile)

**蓝军：** mixed tool + 拼写错误 action（`{"operation":"saev"}`），既不在 ReadOnly 也不在 LocalWrite 也不在 ExternalSend。fallback 把它当 side-effect。**应该是 "unknown action, blocked"。**

if action := structuredAction(ctx.Input); action != "" {

return policyDecision(ToolPolicyAsk, ToolRouteRequiresSideEffectIntent, ctx.Intent.AllowsSideEffects, true, "mixed_side_effect_requires_intent", profile)
```
