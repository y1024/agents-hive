## TL;DR

方向正确：本计划没有新建平行评测平台，而是在现有 `agentquality`、`qualityworkbench`、`master`、`router` 和 Admin surface 上补可信评测闭环。

归档前必须守住的两条红线已经完成：

- 真实 runner 不能先执行外部副作用再事后判违规；现在 `AgentRunEvalRunner` 与 `AgentQualityRunAdapter` 都在生产路径前阻断未声明副作用。
- shadow eval 不能把合成 trace 文本伪装成真实评测输入；现在优先使用事件 attributes 中已脱敏真实 IO，缺失时明确标记 fallback。

## 战略判断

### 1. 计划解决的是质量治理可信度，不是 benchmark 产品

系统已有 candidate、suggestion、approval、rollback、Workbench、Replay、Batch Eval。真正风险不是“没有另一个 benchmark 平台”，而是把不可信 pass 用到审批、上线和回滚。当前实现通过 `RunnerEvidenceLevel` 和审批/apply 门禁把这个风险收住。

### 2. 客服只是试点域，平台抽象不能绑定客服

domain admission 消费统一质量结果：active golden cases、safety cases、runner evidence、judge coverage、semantic score。客服、产品全生命周期、运营、售后等域都应通过同一套 `agentquality + qualityworkbench + domainpolicy` 主线接入，不自建 runner、case store 或审批流。

### 3. 外部 benchmark 是外部可信度，不是业务域上线门禁

benchmark adapter 已标记 `external_benchmark`，并明确不授权 rollout。这个边界合理，避免公开 benchmark 压过真实业务回归集。
