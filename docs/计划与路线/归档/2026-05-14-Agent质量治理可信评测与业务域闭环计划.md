**Goal:** 把现有候选池、优化建议、审批、回滚和 Quality Workbench 从“治理骨架可用”升级为“评测结果可信、可驱动业务域上线”的闭环。

**Architecture:** 不新建平行评测平台，不替换 `agentquality` / `qualityworkbench` / `journal` / `observability` / `router`。在现有质量控制面上补四件事：真实执行 runner、语义 Judge、内部 case 生命周期、可选标准 benchmark adapter。业务域准入只消费统一评测结果，不允许每个业务域自建 runner、case store 或审批闭环。

**Tech Stack:** Go backend (`internal/agentquality`, `internal/qualityworkbench`, `internal/master`, `internal/router`, `internal/memory`, `internal/api`), PostgreSQL stores, React admin UI, existing LLM router, table-driven Go tests, focused frontend build/tests when UI changes.

## 0. 结论

外部评价的大方向是对的，但系统要解决的问题不是“补一个完整 benchmark 产品”，而是“让质量治理闭环里的决策可信”。

| 评价项 | 系统判断 | 本计划处理 |
|---|---|---|
| 架构骨架很强 | 成立 | 保留现有 `agentquality + qualityworkbench + replay/batch eval + approval/rollback` 主线 |
| 候选池 + 建议 + 审批 + 回滚是亮点 | 成立 | 继续作为产品核心，不另建业务域审批系统 |
| 默认 Pass 不可信 | 成立，且是主风险 | P0 接真实 runner，静态 runner 降级为 dry-run/schema-check |
| 没有 LLM-as-Judge | 基本成立 | P1 增加 Judge 接口、rubric、校准集和审计输出 |
| case 全靠手写，没有标准 benchmark | 基本成立 | 先建内部 case 生命周期，公开 benchmark 做 adapter，不进主路径 |

当前最危险的不是缺少公开 benchmark，而是把不可信的 `Pass` 用到优化建议、审批、上线和回滚判断里。治理链条越完整，错误评测的放大效应越强。

## 1. 现状基线

### 1.1 已经具备的能力

- `internal/agentquality` 已有 case schema、gate metrics、candidate、suggestion、approval、rollback、rollout 和 eval diff。
- `internal/qualityworkbench` 已有 replay job、batch eval、cluster、dashboard、weekly report 和 PG/memory store。

- `internal/master/quality_events.go` 已经开始把质量事件打入 metric/log/journal，并有 redaction 草案。
- `internal/domainpolicy` 已经把业务域准入从 `runtimepolicy` 中分离出来，方向正确。

这些说明系统不缺“质量治理骨架”。下一步不应重建平台，而应提高评测结果的真实性。

### 1.2 当前主要缺口

| 缺口 | 代码证据 | 风险 |
|---|---|---|
| 默认 eval runner 是静态 Pass | `internal/agentquality/eval_runner.go` 的 `StaticEvalRunner` 对每个 case 写 `Passed: true` | Pass 只能证明 schema/gate plumbing，不证明 agent 做对 |
| Batch eval / replay 依赖外部 runner 注入 | `qualityworkbench.BatchEvalStart.EvalRunner` 和 `ReplayRunner.EvalRunner` | 没有生产 runner 时结果 unknown 或静态假阳性 |
| 语义 evaluator 是启发式 | `HeuristicReflectionEvaluator` 按 trigger 返回固定 verdict | 不能判断回答是否真正解决用户问题 |
| case schema 偏结构化 | `agentquality.Case` 主要描述 input、expected tools、allowed tools、status | 缺少 expected answer、judge rubric、assertions、domain metadata |
| 内部 case 生命周期不完整 | 当前有手写 `aq01`-`aq10` 和 candidate golden case 导出 | 线上失败、人工反馈、回滚事故没有稳定晋升为 regression corpus |
| 标准 benchmark 没接入执行主线 | 只有研究/调研材料，未见可运行 adapter | 对外可信度和横向比较不足，但不是近期主阻塞 |

## 2. 行业实践映射

本计划采用行业实践的原则，不引入新平台依赖。

| 行业实践 | 可采纳原则 | Hive 映射 |
|---|---|---|
| OpenAI Evals / graders | eval 要用真实输入、明确 grader/rubric、把结果作为模型/agent 变更门禁 | `EvalRunner` 跑真实 session，`JudgeEvaluator` 输出结构化 verdict |
| OpenAI agent tracing/evals | agent 行为需要 trace、tool call、final output 一起评估 | 复用 Journal / Trace / Quality Event / RouteDecision |
| LangSmith datasets / online evals | 生产 trace 和人工反馈应回流到 dataset，离线回归与线上抽样结合 | candidate -> triage -> golden case -> regression suite -> shadow eval |
| OpenTelemetry | metric label 避免高基数；高基数 identity 留在 trace/log/document field | `MetricLabels` 禁止 `session_id/user_id/run_id/trace_id`，诊断字段进 trace/log |
| OWASP LLM Top 10 | 防 prompt injection、敏感信息泄漏、过度代理能力和不安全输出处理 | redaction、capability gate、HITL、tool policy、Judge 输入脱敏 |

参考资料：

- OpenAI Evals / graders: https://platform.openai.com/docs/guides/evals
- OpenAI Agent Evals / tracing: https://platform.openai.com/docs/guides/agent-evals
- LangSmith Evaluation concepts: https://docs.langchain.com/langsmith/evaluation-concepts
- OpenTelemetry semantic conventions: https://opentelemetry.io/docs/specs/semconv/
- OWASP Top 10 for LLM Applications 2025: https://genai.owasp.org/resource/owasp-top-10-for-llm-applications-2025/

## 3. 目标状态

### 3.1 质量结果分级

所有 eval 结果必须带 `evidence_level`，禁止把不同可信度的 Pass 混用。

| evidence_level | 含义 | 是否允许驱动审批/上线 |
|---|---|---|
| `static_schema` | 只验证 case schema、gate pipeline、report 生成 | 不允许 |
| `replay_trace` | 基于已有 trace/journal 重放和结构化断言 | 仅允许发现问题，不允许证明修复 |
| `simulated_runner` | 用 fake tool / fixture runner 跑真实 agent loop | 可用于 PR gate |
| `real_runner` | 跑真实 master/session/tool path，外部副作用 sandbox/mock | 可用于优化上线门禁 |
| `production_shadow` | 线上只读 shadow / 抽样 judge，不影响用户响应 | 可用于监控和回滚 |
| `human_verified` | 人工复核确认 | 可用于高风险审批和黄金集晋升 |

### 3.2 闭环主链路

```text
线上质量事件 / 用户反馈 / 人工审批拒绝 / 回滚事故
        ↓
Candidate
        ↓
Triage: 去重、归因、脱敏、业务域标注
        ↓
Golden Case Draft
        ↓
Runner: static schema + simulated/real execution
        ↓
Judge: 规则断言 + LLM rubric + 人工抽检
        ↓
Gate: domain regression / safety / performance / semantic score
        ↓
Suggestion / Approval / Rollout / Rollback
        ↓
Case 晋升或废弃
```

### 3.3 不做什么

- 不新建独立 eval SaaS。
- 不把公开 benchmark 作为近期主门禁。
- 不让业务域自建 case store、runner、judge 或审批流。
- 不让 LLM-as-Judge 单独决定高风险上线。
- 不把 raw trace、raw user input、token、cookie、工具参数原文送入 Judge。
- 不把 `runtimepolicy` 扩成业务域授权或评测策略。

## 4. 核心抽象

### 4.1 EvalRunner 分层

保留现有接口：

```go
type EvalRunner interface {
    Run(cases []LoadedCase) (GateInput, error)
}
```

新增 runner metadata，避免破坏已有调用方：

```go
type RunnerEvidenceLevel string

const (
    EvidenceStaticSchema     RunnerEvidenceLevel = "static_schema"
    EvidenceReplayTrace      RunnerEvidenceLevel = "replay_trace"
    EvidenceSimulatedRunner  RunnerEvidenceLevel = "simulated_runner"
    EvidenceRealRunner       RunnerEvidenceLevel = "real_runner"
    EvidenceProductionShadow RunnerEvidenceLevel = "production_shadow"
)

type RunnerInfo struct {
    Name          string              `json:"name"`
    Version       string              `json:"version"`
    EvidenceLevel RunnerEvidenceLevel `json:"evidence_level"`
}

type DescribedEvalRunner interface {
    EvalRunner
    Info() RunnerInfo
}
```

规则：

- `StaticEvalRunner.Info().EvidenceLevel` 必须是 `static_schema`。
- `BatchEvalRun`、`ReplayJobResult`、`GateInput` 或 result metadata 必须记录 runner info。
- `static_schema` 结果不允许生成“可上线”建议，只能生成“case/gate plumbing 可用”结论。

### 4.2 AgentRunEvalRunner

新增真实 runner，但不新建 `internal/run`。

建议位置：

- `internal/agentquality/agent_runner.go`
- `internal/agentquality/agent_runner_test.go`

职责：

- 按 case 构造隔离 session。
- 调用现有 master/session 入口或最小可注入的 agent execution adapter。
- 捕获 `RouteDecision`、tool call、quality event、journal decision、trace span、final output。
- 对危险工具使用 sandbox/fake invoker，不执行真实外部写入。
- 产出 `GateInput`，包含结构化 events、results、tool actual、replay refs。

最小接口：

```go
type AgentRunAdapter interface {
    RunCase(ctx context.Context, input AgentRunCaseInput) (AgentRunCaseOutput, error)
}

type AgentRunCaseInput struct {
    Case      Case
    DomainID  string
    OwnerID   string
    SessionID string
}

type AgentRunCaseOutput struct {
    FinalOutput string
    FinalStatus FinalStatus
    Events      []Event
    TraceID     string
    ReplayRef   string
    ToolCalls   []ObservedToolCall
}
```

### 4.3 JudgeEvaluator

保留 `EvaluationInput` / `EvaluationVerdict`，新增 judge 维度。

```go
type JudgeInput struct {
    CaseID          string            `json:"case_id"`
    DomainID        string            `json:"domain_id,omitempty"`
    UserInput       string            `json:"user_input"`
    FinalOutput     string            `json:"final_output"`
    ExpectedAnswer  string            `json:"expected_answer,omitempty"`
    Rubric          []RubricCriterion `json:"rubric,omitempty"`
    TraceSummary    TraceSummary      `json:"trace_summary,omitempty"`
    ToolAssertions  []ToolAssertion   `json:"tool_assertions,omitempty"`
}

type RubricCriterion struct {
    Name        string `json:"name"`
    Description string `json:"description"`
    Weight      int    `json:"weight"`
}

type JudgeVerdict struct {
    Score          int         `json:"score"`
    Passed         bool        `json:"passed"`
    FailureType    FailureType `json:"failure_type,omitempty"`
    Rationale      string      `json:"rationale"`
    Evidence       []string    `json:"evidence,omitempty"`
    ShouldOptimize bool        `json:"should_optimize"`
}
```

规则：

- Judge 输入必须先经过 redaction。
- Judge 输出必须通过 schema validation。
- Judge prompt/version/model 必须记录到事件或 run metadata。
- Judge 只参与评分，不绕过 deterministic safety gate。
- 高风险业务域上线需要 deterministic gate + Judge score + 人工审批。

### 4.4 Case schema 扩展

现有 `Case` 保持兼容，新增可选字段：

```go
type Case struct {
    // existing fields...
    DomainID         string            `json:"domain_id,omitempty"`
    SourceKind       string            `json:"source_kind,omitempty"`
    SourceName       string            `json:"source_name,omitempty"`
    ExpectedAnswer   string            `json:"expected_answer,omitempty"`
    Assertions       []CaseAssertion   `json:"assertions,omitempty"`
    JudgeRubric      []RubricCriterion `json:"judge_rubric,omitempty"`
    EvidenceLevelMin string            `json:"evidence_level_min,omitempty"`
    Tags             []string          `json:"tags,omitempty"`
    CreatedFrom      string            `json:"created_from,omitempty"`
}
```

断言类型首批只做必要集合：

| assertion | 含义 |
|---|---|
| `tool_called` | 必须调用某工具 |
| `tool_not_called` | 禁止调用某工具 |
| `status_is` | final status 必须匹配 |
| `answer_contains` | 输出必须包含关键事实 |
| `answer_not_contains` | 输出不得包含敏感/错误内容 |
| `requires_clarification` | 必须提出澄清 |
| `no_external_write` | 不允许外部写入 |
| `trace_has_event` | trace/quality event 必须出现 |

## 6. 验收门禁

### 6.1 PR 门禁

- 新增或修改 router/tool/memory/master 行为时，必须跑相关 domain regression suite。
- 如果只跑 `static_schema`，PR 只能声明“schema/gate 检查通过”，不能声明质量通过。
- 涉及外部写入、权限、HITL、业务域准入，必须有 safety cases。

### 6.2 优化建议门禁

优化建议进入审批前必须满足：

- 有真实 runner 或 production shadow 证据。
- 有 deterministic gate 结果。
- 涉及语义质量时有 Judge verdict。
- 涉及高风险工具时有人审记录。
- 能回溯到 case ids、trace refs、prompt/model versions。

### 6.3 业务域上线门禁

业务域从 disabled 到 enabled 必须满足：

- active golden cases 达到该 domain 最小数量。
- 包含 allow/deny/clarification/safety/regression 五类 case。
- 最近一次 regression suite 不是 `static_schema`。
- semantic score 达标。
- 没有 P0/P1 safety failure。
- Workbench 能按 domain/source 归因。

## 7. 推荐实施顺序

1. **先做 Phase 0**：立刻阻止静态 Pass 被误用。
2. **再做 Phase 1**：接真实 runner，这是所有可信评测的根。
3. **并行准备 Phase 3 的 case 生命周期**：先把 candidate -> golden draft 打通。
4. **Phase 2 Judge 跟随 runner 落地**：先 fake/heuristic 校准，再接 LLM judge。
5. **Phase 4 让业务域消费 regression suite**：客服试点只在门禁通过后开启。
6. **Phase 5 做生产 shadow**：用于持续监控和回滚。
7. **Phase 6 最后做标准 benchmark adapter**：用于对外展示，不抢主线。
