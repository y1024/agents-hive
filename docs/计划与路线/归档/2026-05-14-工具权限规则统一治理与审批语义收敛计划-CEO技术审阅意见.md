## 一、CEO 视角：根问题诊断

### 1.1 根问题不是「4 维语义需要拆」，根问题是「同一字段被复用成多种意思」

计划 Section 0 把 `requires_approval` 拆成 Capability / RiskClass / RouteAdmission / ExecutionApproval 四维。方向是对的，但比真实痛点大一号。

用户实际痛点（计划开头 line 5 列的 3 件事）：

| 痛点 | 根因 |
|---|---|
| 飞书发文本被 HITL 卡住 | `feishu_api.send_message` 分类 = `ExternalSendActions`，路由层把整个外发集合等同 ask |
| MCP 已连接但模型说 callable_now=false | `tool_visibility.go:505` 把 `policy.RequiresApproval` OR `policy.Action==Deny` 推导成 MayRequireApproval，污染了 catalog |
| discovery_only 自我拒绝 | `tool_search.go:102` 同样的 OR 把展示层污染 |

**只有第一件是真的「需要新分类」**。第 2、3 件是 4 个消费点没共用同一份 `policy.MayRequireApproval` 投影 — 不需要新概念，需要的是收敛 5 个消费点。

### 1.2 概念模型存在 over-engineering 风险

计划新增 4 个维度，但其中 2 个完全等价于现有结构：

| 计划新增 | 现状已有 | 真实差距 |
|---|---|---|
| `RouteAdmission.callable_now` | `ToolPolicyDecision.CallableNow` (`tool_policy.go:39`) | 等价 |
| `RouteAdmission.allowed_inputs` | `RouteDecision.AllowedToolInputs` (`decision.go:46`) | 等价 |
| `ExecutionApproval{none, required, denied}` | `ToolPolicyAction{Allow, Ask, Deny}` (`tool_policy.go:11-14`) | 几乎等价（区别只是 ask 是否一定要 HITL channel）|
| `RiskClass{read_only / routine_side_effect / privileged / destructive / runtime_exec / unknown}` | `RiskLevel{ReadOnly / LocalWrite / ExternalWrite / RuntimeExec / Destructive / Unknown}` | **routine_side_effect / privileged 是新增**，是真正的差距 |

**结论**：4 维模型里只有「细化 ExternalWrite 为 routine vs privileged」是真新增。新增独立 `Approval` 字段与 `Action` 重复存储同一信息（计划 Task 3.1 没说两者关系是 derived 还是独立来源 — 这是设计缺口）。

**建议**：删 `Approval`，保留 `Action`。新增的两个真正有价值的字段是：

- `RiskClass`（事实层标签，用于日志、审计、UI 提示）
- `MayRequireApproval`（catalog/admission hint，与 `RequiresApproval` 显式分离）

### 1.3 单 PR 改动面不可接受

计划 9 大 Task × 7 module × Go 后端 + React 前端 + Postgres migration + 文档。任何一项失败都把整个 PR 卡住。必须拆三段：

| Phase | 目标 | Task 范围 | 估时（人工） | 估时（CC+gstack） |
|---|---|---|---|---|
| **A 止血** | 3 件用户痛点全部解决 | Task 2 (routine 分类) + Task 5.1/5.3 (ActionGuard) + Task 6.2 (mcp summary 拆字段) | 1 天 | 1-2 小时 |
| **B 语义收敛** | 概念模型落代码 | Task 3 (新字段) + Task 4 (decision 简化) + Task 6.1/6.3 + Task 7 (UI 文案) | 1-2 天 | 2-4 小时 |
| **C 治理** | 文档/可观测/回归 | Task 8 (日志) + Task 9 (文档) + 验收矩阵 + metric | 0.5-1 天 | 1-2 小时 |

**Phase A 单独 ship 就能止住生产 user-visible bug**。B/C 可走灰度，错了影响展示和审计而不是行为。

### 1.4 策略与配置的边界争议未解决

Task 7 要把 WebUI 「权限规则」改成 「Legacy HITL Rules」。方向对，但计划没回答：

- 目前生产 `hitl.permission_rules` 是否有客户在配置？
- 如果有，UI 文案变更是行为变化还是纯文案？
- 文案变更前后客户配置是否还兼容？

**建议**：Task 7 之前先跑一次 `SELECT count(*) FROM config WHERE jsonb_path_exists(value, '$.hitl.permission_rules')`，按结果决定文案是「Legacy HITL Rules」还是「Legacy HITL Rules (deprecated)」。零客户用 = 直接 deprecate；有客户用 = 保留 + 加迁移指引。

## 二、顶级技术专家视角：按 Severity 排序的发现

### CRITICAL-1：`tool_policy_test.go:152-167` 的 case 是构造的 unreachable state，锁死了错误契约

```go
{
    name:         "unclassified deny still requires attention",
    def:          mcphost.ToolDefinition{Name: "question", Core: true},
    intent:       IntentFrame{Kind: IntentAnswer},
    input:        json.RawMessage(`{}`),
    wantAction:   ToolPolicyDeny,
    wantRoute:    ToolRouteBlockedUnknown,
    wantApproval: true,                          // ← 锁死 RequiresApproval=true
},
...
if tt.name == "unclassified deny still requires attention" {
    profile.ReadOnly = false                     // ← 手动 mutate profile
    profile.Risk = RiskReadOnly
}
```

这个 case 通过手动改 profile，让它走到 `tool_policy.go:84` 的 `unclassified_tool_policy` fall-through 分支，并 assert `RequiresApproval=true`。这把「fall-through deny 必须 RequiresApproval=true」锁成回归契约。

而 Task 3 的目标恰恰是把 `RequiresApproval` 收窄成「当前这次执行必须人工确认」。fall-through deny 是 router 决策失败，不是「执行需要人确认」 — 这个 case 直接破坏了 Task 3 的整个语义。

**修法**：删此 case 或重写成「unreachable，记录 `Reason` 日志即可」，不要 assert `RequiresApproval`。

### CRITICAL-2：`action_guard.go:73, 107` 已有的 `plain_text_im_send` 硬编码与 Task 5.1 方向冲突

```go
// internal/master/action_guard.go:73-75 和 107-109
if isPlainTextIMSendAction(toolName, input.Arguments) {
    return actionGuardDecision(ActionGuardAllow, "plain_text_im_send", "tool_policy", "")
}
```

计划 Task 5.1 写：「ActionGuard 必须不含 `if toolName == "send_im_message" { return ask }` 类似逻辑」。

**但当前代码已有相反方向的硬编码：直接 allow plain text IM send**。这是之前为了止血用户痛点临时打的补丁。

计划 Section 0.1 说「不回滚用户或其他 agent 的修改」，但完全没识别这条已存在的硬编码。如果 Task 5.1 实施时按字面执行（删所有外发硬编码），会把这个止血补丁也删掉 — 飞书发文本立刻再次卡死。

**修法二选一**：

| 方案 | 优点 | 缺点 |
|---|---|---|
| **A. 删硬编码 + 完整依赖 Task 2 routine 分类** | 语义干净，单一信源 | 必须先把 routine 分类 100% 实现并测试到位，才能删硬编码，否则瞬间退化 |
| **B. 保留硬编码作为 fast path + 注释「Task 2 routine 完成后可删」** | 增量安全 | 多了一处分支逻辑 |

**推荐 B**：作为 Phase A 止血方案的兜底，先保留；Phase B 完成 Task 2 + consistency test 后再删。

### CRITICAL-3：`action_guard.go:187` 的 dead call

```go
// internal/master/action_guard.go:187
return router.StructuredDangerousAction("tool_policy", action) || toolActionLooksDangerous(action)
```

`structuredDangerousActions` map 里没有 `"tool_policy"` 这个 key（见 `capability_registry.go:133-144`）。这个调用永远返回 `false`。

整段 `toolValueRequiresApproval` 检测「参数里包含危险 action 关键字」的能力，实际只靠 `toolActionLooksDangerous` 关键字匹配（`delete/drop/truncate/...`），第一个分支是 dead code。

**这是潜在 bug**，计划完全没提到。Task 5 重写 ActionGuard 时应当顺手修，并加 unit test 覆盖「插件传 `{"action":"delete_user"}` 给非 mixed 工具时被识别为危险」。

### HIGH-1：四维模型 vs 现状代码的真实落差被高估

计划 1.1 表格说「`RequiresApproval` 同时被用作 catalog 风险和当前执行审批」。这个诊断对，但代码里**只有 4 处真正污染**：

| 2 | `tool_search.go:102` | `policy.RequiresApproval \|\| policy.Action == ToolPolicyDeny` | 拆成 `RequiresApproval` 用执行投影，`MayRequireApproval` 用 catalog 投影 |
| 3 | `tool_visibility.go:505-508` | 同样的 OR | 同上 |

### HIGH-2：`mixedActionRule` 现状已经接近计划「目标」，差距被高估

计划 Task 2.1 给的「当前结构」（line 351-354）：

```go
type mixedActionRule struct { Field, ReadOnlyActions, LocalWriteActions, ExternalSendActions }
```

`capability_registry.go:151-156` 实际：

```go
type mixedActionRule struct { Field, ReadOnlyActions, LocalWriteActions, ExternalSendActions }
```

**完全一致**。差距只在两件具体的事：

1. `feishu_api` 的 `ExternalSendActions` 没区分 `send_message`（routine） vs `send_file/upload_file`（privileged） vs `create_task`（dangerous，已在 structuredDangerousActions）
2. 缺一个 `RoutineSideEffectActions` 字段把 routine 显式标出来

**修法**：

```go
type mixedActionRule struct {
    Field                    string
    ReadOnlyActions          []string
    RoutineSideEffectActions []string  // 新增
    LocalWriteActions        []string
    ExternalSendActions      []string  // 保留作路由约束 list（用于 AllowedToolInputs）
}
```

`feishu_api.send_message` 同时存在于 `RoutineSideEffectActions` 和 `ExternalSendActions`：
- `RoutineSideEffectActions` 决定 ApprovalRequired=false
- `ExternalSendActions` 决定 AllowedToolInputs（路由仍只在 external write intent 下放行）

改动量：~30 行 Go + 5 个测试 case，不是计划 Task 2 描述的「扩展 classifier 基础设施」那种规模。

### HIGH-3：trusted_remote 的 sanitize 链路是隐藏安全 anchor，必须显式契约化

`profile_infer.go:303-345` `markProfileSanitizeBlocked` 是当前最强的 fail-closed 防线：

- description 含 prompt injection 关键字 → profile 立即降级 `unknown/destructive/openworld`
- 然后 `tool_policy.go:50-56` 直接 deny

这条链路对 LLM 注入攻击是关键防御。计划 Task 7.3 写「MCP server trusted 只代表 catalog 来源可信」，方向对，但**没把这条 sanitize fail-closed 提升为契约**。

如果未来某次重构有人改 `markProfileSanitizeBlocked` 让 trusted 工具豁免 sanitize（"反正 trusted 嘛"），整个 LLM 注入防御就破了。

**修法**：

1. Task 7.3 加一条契约：「Trusted-status MUST NOT bypass description/schema sanitize. Sanitize-blocked profile 永远 fail-closed，无论 source 是否 trusted」
2. 加测试 `TestTrustedRemoteWithInjectedDescriptionStillFailsClosed`，构造 trusted + description 含注入关键字的 case，assert profile 被降级且 EvaluateToolPolicy 返回 Deny

### MEDIUM-1：`feishu_api.send_message` 的 routine 边界没定义

### MEDIUM-2：缺 contract test 覆盖「同一工具在所有消费点结论一致」

计划 Section 9 「完成定义」第 4 条说：「`tool_search`、`mcp.tools.list`、`tool_visibility`、`RouteDecision`、`ActionGuard` 对同一工具给出一致解释」。

但计划**没有 contract test 强制它们消费同一份策略输出**。未来很容易再次 drift（这就是当前 bug 的根因）。

**修法**：新建 `internal/router/policy_consistency_test.go`，对一组黄金工具 × 输入矩阵跑：

```go
func TestPolicyConsistencyAcrossConsumers(t *testing.T) {
    fixtures := []struct {
        def    mcphost.ToolDefinition
        intent IntentFrame
        input  json.RawMessage
        wantRequiresApproval bool
        wantMayRequireApproval bool
        wantCallable bool
        wantActionGuard string // allow/ask/deny
    }{
        // 至少 30 个 golden case 覆盖：
        // - read_file / grep / websearch (read_only)
        // - im_api.send_message text (routine)
        // - feishu_api.send_message text (routine)
        // - feishu_api.send_file (privileged)
        // - feishu_api.create_task (dangerous)
        // - memory.search / memory.delete
        // - taskboard.create / taskboard.delete
        // - bash pwd / bash rm -rf
        // - trusted MCP read / trusted MCP side-effect / trusted MCP destructive
        // - untrusted MCP
        // - feishu_api on wechat intent (should NOT satisfy)
    }
    for _, f := range fixtures {
        profile := router.InferToolProfile(f.def, ...)
        policy := router.EvaluateToolPolicy(profile, ...)

        // 5 个消费点全部 assert 同一结论
        assertToolSearchOutput(t, f, profile, policy)
        assertMCPSummaryOutput(t, f, profile, policy)
        assertToolVisibilityAdmission(t, f, profile, policy)
        assertRouteDecisionCallable(t, f, profile, policy)
        assertActionGuardDecide(t, f, profile, policy)
    }
}
```

**这是这次重构最值钱的产出物**，比任何概念模型都重要。是防止再次 drift 的唯一硬保险。

## 五、CEO 最终判断

### 5.1 方向

**正确**。把权限语义从「一个字段多义」收敛到「事实层 + 决策层 + 展示层」分离，是这个系统长期可维护性的关键改造。`policy_consistency_test` 黄金矩阵是这次重构最有价值的工程资产。

### 5.2 节奏

**计划当前节奏太激进**。9 Task × 7 module × 1 PR 是高风险动作。必须拆 Phase A/B/C 三段 ship，Phase A 优先解决用户痛点。

### 5.3 概念模型

**4 维过头**。删 `ExecutionApproval`，保留 3 维：`RiskClass`（事实）+ `Action`（决策）+ `MayRequireApproval`（展示）。代码工作量降 ~40%。

### 5.4 最大隐藏风险

`action_guard.go` 已有的 `plain_text_im_send` 硬编码与 Task 5.1 描述的目标方向反着，计划完全没识别。这是 single point fragile。Phase A 必须显式处理（推荐保留 + 注释），否则盲改会瞬间退化。

### 5.5 必须的契约化

Trusted 与 sanitize 的关系必须显式契约化：「trusted-status MUST NOT bypass description/schema sanitize」。这是 LLM 注入下唯一守不住的地方。计划 Task 7.3 提到了 trusted 的边界，但没把这条提升为一等位的安全契约。

### 5.6 验收和回滚

`policy_consistency_test` 黄金矩阵 + strict 路径回归 + dual-write 14 天 + 4 个生产 metric — 这四件事是 production-grade 上线的最低门槛。计划目前都缺。
