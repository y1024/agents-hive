# 工具权限规则统一治理与审批语义收敛计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use `executing-plans` or equivalent task-by-task execution. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 把工具召回、路由、展示、执行守卫、HITL 审批和 WebUI 配置统一到同一套权限语义与同一套工具抽象，根治“工具已召回但不可用”“普通 IM 文本发送被当危险动作审批”“MCP/内置/自定义工具各自为战”的问题。

**Architecture:** 建立“Provider/Descriptor -> 事实画像 -> 意图路由 -> 策略裁决 -> Runtime 执行守卫 -> 审计展示”的单向链路。`ToolProvider` 统一 MCP、builtin、Skill、IM、bash 和自定义工具来源；`ToolDescriptor` 是所有消费点共享的事实输入；`ToolProfile` 只表达可信事实，`ToolPolicyDecision` 只表达当前上下文裁决，`ActionGuard` / Runtime 只执行同一裁决和少量命令级补充检查，WebUI 只能配置运行时开关和集成参数，不能绕过 host-owned 安全边界。

**Tech Stack:** Go backend (`internal/router`, `internal/master`, `internal/tools`, `internal/gateway`, `internal/config`, `internal/store`, `internal/security`), React WebUI (`frontend/src/components/settings`), table-driven Go tests, focused npm tests/build only if UI copy or API types change.

## 0. CEO / 架构判断

当前问题不是某一个工具的 bug，而是权限语义被多处复用成了不同意思：

- `requires_approval` 有时表示“这次执行必须人工确认”。
- 有时又表示“这个工具有副作用”。
- 有时还表示“这个工具未知/危险/当前不可调用”。
- `tool_search`、`mcp.tools.list`、`tool_visibility`、`RouteDecision`、`ActionGuard` 各自消费这个字段时语义不完全一致。

这会直接导致产品错觉：工具已经被召回、MCP 已连接、列表里也能看到，但模型仍回复“discovery_only / requires_approval / callable_now=false”，或者普通飞书文本发送被 HITL 卡住。

本计划的核心改法有两层：

1. 工具来源必须统一成可插拔接口。后续新接 MCP server、企业 IM、内置工具、plugin tool、Skill workflow，只能实现 provider/descriptor/invoker adapter，不能再把工具类别硬编码进展示层、召回层和执行层。
2. 权限拆成三个不混用的维度，并保留现有 `ToolPolicyAction` 作为唯一执行裁决：

| 维度 | 含义 | 例子 | 允许谁改 |
|---|---|---|---|
| `Capability` | 工具具备什么能力 | `external.send.feishu`, `runtime.exec` | 代码审查 |
| `RiskClass` | 动作客观风险 | `read_only`, `routine_side_effect`, `privileged_side_effect`, `destructive`, `runtime_exec`, `unknown` | 代码审查 |
| `RouteAdmission` | 当前用户意图下能否给模型调用 | `callable_now`, `allowed_inputs`, `blocked_reason` | 策略代码 |

执行层不新增第二套 `ApprovalDecision`。`ToolPolicyAction{allow, ask, deny}` 已经表达“放行 / 需要执行前确认 / 拒绝”，新增 `Approval` 字段会和 `Action` 重复并引入一致性风险。

其中 `requires_approval` 只能保留为 **`Action == ask` 且当前这次执行必须人工确认** 的布尔投影。如果只是“该工具有些危险动作”，必须用 `may_require_approval` / `dangerous_actions` / `risk_class` 表达。

## 0.1 统一工具抽象合同

新增或等价实现以下平台合同，允许先用 adapter 包住现有 `mcphost.Host`，但所有新工具类型必须接这个抽象：

```go
type ToolDescriptor struct {
    Definition mcphost.ToolDefinition
    Profile    router.ToolProfile
    Entry      router.CapabilityEntry
}

type ToolProvider interface {
    ListToolDescriptors(context.Context) ([]ToolDescriptor, error)
    LookupToolDescriptor(context.Context, string) (ToolDescriptor, bool)
}

type ToolInvoker interface {
    InvokeTool(context.Context, ToolInvocation) (*mcphost.ToolResult, error)
}

type ToolInvocation struct {
    Name      string
    Arguments json.RawMessage
    Intent    router.IntentFrame
    Route     router.RouteDecision
}
```

迁移规则：

- `tool_search`、`mcp.tools.list`、`tool_visibility`、`RouteDecision`、`ActionGuard` 必须以 `ToolDescriptor/Profile/PolicyDecision` 为输入，不再各自推断风险语义。
- `Master` 主会话、sub-agent `ToolBridge`、direct execution 都必须在执行前经过同一个 runtime gate；插件改写参数后必须重新检查。
- `SafeExecutor` 只作为 `runtime.exec` 的命令级补充边界，不承担通用工具策略。
- `PermissionManager` / legacy HITL 只能是 `strict` 模式或 runtime ask 的执行通道，不能作为另一套授权来源。
- 新增工具来源只能新增 provider adapter；除非引入新的能力或风险类别，否则不能修改核心策略层分支。

## 1. 当前规则和权限到底在哪里

### 1.1 Host-owned 代码规则

这些是安全边界，必须通过代码审查和测试修改，不能开放成 WebUI 任意配置。

| 责任 | 当前文件 | 当前问题 | 收敛方向 |
|---|---|---|---|

| profile 推断 | `internal/router/profile_infer.go`, `internal/router/tool_profile.go` | `TrustedRemoteToolProfile` 默认 read-only，但写入启发式与执行策略耦合 | 只产出可信事实与风险分类 |

| 执行守卫 | `internal/master/action_guard.go`, `internal/master/action_guard_runtime.go` | ActionGuard 仍存在外发/参数/危险扫描的并行判断 | ActionGuard 主体只消费统一 execution decision，SafeExecutor 只管 shell |
| legacy 权限管理 | `internal/master/lifecycle.go`, `internal/skills/permission.go` | `permission_rules` 与新 ActionGuard 并存，容易双审批或错审批 | strict 回滚保留，minimal 主路径走统一策略 |
| shell 安全 | `internal/security/builtin_rules.go`, `internal/security` | 这是独立命令级安全边界 | 保留，挂在 runtime exec 的补充检查 |

### 1.2 WebUI / 运行时配置入口

这些入口只能改运行时参数，不能当成最终安全策略。

| 配置 | 后端入口 | 前端入口 | 定位 |
|---|---|---|---|
| `hitl.enabled` | `internal/gateway/methods_config.go` -> `config.update` | `frontend/src/components/settings/PermissionRulesSettings.tsx` | 是否启用人工审批通道 |
| `hitl.permission_rules` | `methods_config.go`, `internal/store/postgres_migrate.go` | `PermissionRulesSettings.tsx` | legacy/strict 权限规则；不应决定普通工具是否 routable |
| `agent.action_guard_enabled` | `internal/config/defaults.go`, `internal/store/postgres_migrate.go` | 当前无清晰 UI | 是否启用执行守卫 |
| `security.permission_mode` | `internal/config/config.go`, `internal/master/lifecycle.go` | 安全配置页 | `minimal` 主路径，`strict` 回滚 |
| MCP server transport/header/env | `methods_config.go`, `internal/store/postgres_migrate.go` | `frontend/src/components/settings/MCPServersSettings.tsx` | 只配置连接；不等于授权 |
| MCP/tools summary | `internal/gateway/methods_mcp.go` | `MCPServersSettings.tsx` 工具目录 | 只能展示 policy 结果，不能自行推断 |

结论：WebUI 上“权限规则”页现在改的是 legacy HITL/PermissionManager，不是统一工具策略层。普通 IM 文本发送被拦住，不能靠在 WebUI 把某个工具改成 allow 来根治，必须在统一策略层修正。

## 2. 产品原则

1. **发现不是授权**：`tool_search` / MCP catalog 只能告诉模型有什么工具，不直接授权执行。
2. **授权来自 RouteDecision**：本轮能不能调用只能看 `RouteDecision.AllowedTools` 和 `AllowedToolInputs`。
3. **执行再过 ActionGuard**：工具调用参数可能被插件或模型改写，执行前必须按同一策略再验一次。
4. **普通动作默认不审批**：只要用户意图明确、平台匹配、输入完整、作用域合法，普通文本 IM 发送、只读查询、常规本地工作流动作不应弹 HITL。
5. **危险动作才审批或拒绝**：删除、开放世界执行、shell 危险命令、未知工具、生产写库、媒体/文件外发、发布/审批/任务创建等高风险动作才进入审批或 deny。
6. **缺参数不是审批**：缺 recipient、缺 content、平台不明确，是澄清问题，不是 HITL 审批问题。
7. **trusted 只信来源，不信动作**：MCP server 被配置成功或被标记 trusted，只代表 catalog 来源可信；具体工具动作仍按风险和意图判定。
8. **策略要可解释**：日志、tool_search、MCP 工具目录、质量事件必须输出同一套 reason，不能前端显示“需要审批”而执行层实际 allow。

## 4. 新模型

### 4.1 RiskClass

在 `internal/router/tool_policy.go` 或新文件 `internal/router/tool_risk.go` 引入统一风险分类：

```go
type ToolRiskClass string

const (
	ToolRiskReadOnly             ToolRiskClass = "read_only"
	ToolRiskRoutineSideEffect    ToolRiskClass = "routine_side_effect"
	ToolRiskPrivilegedSideEffect ToolRiskClass = "privileged_side_effect"
	ToolRiskDestructive          ToolRiskClass = "destructive"
	ToolRiskRuntimeExec          ToolRiskClass = "runtime_exec"
	ToolRiskUnknown              ToolRiskClass = "unknown"
)
```

映射原则：

- `read_only`: 不改变外部/本地状态。
- `routine_side_effect`: 用户明确要求后可直接执行的常规动作，例如普通文本 IM 发送、todo 创建、workspace 内文件编辑。
- `privileged_side_effect`: 会影响外部用户、团队系统、文件/媒体、审批、发布、生产环境的动作。
- `destructive`: 删除、覆盖不可恢复资源、drop/truncate、remove tool。
- `runtime_exec`: shell/exec/run_command。
- `unknown`: 无可信画像或无法判断。

### 4.2 Decision Projection

不新增独立 `ApprovalDecision`。审批语义由现有 `ToolPolicyAction` 投影：

```go
RequiresApproval = decision.Action == ToolPolicyAsk && decision.CallableNow
MayRequireApproval = profile/action has privileged, destructive, runtime, or unknown branches
```

`ToolPolicyDecision` 增加或明确这些字段：

```go
type ToolPolicyDecision struct {
	Action                   ToolPolicyAction
	RouteStatus              ToolRouteStatus
	CallableNow              bool
	RequiresApproval         bool // only approval required for this concrete context
	MayRequireApproval       bool // catalog/admission hint only
	RequiresSideEffectIntent bool
	RiskClass                ToolRiskClass
	Reason                   string
	Source                   string
	ReadOnly                 bool
	SideEffect               bool
}
```

兼容要求：

- 对外 JSON 可继续返回 `requires_approval`，但语义必须收窄为“当前执行需审批”。
- 新增 `may_require_approval` 给 tool_search / MCP tools list / admission observation 用。
- 双写迁移期间旧 `requires_approval` 字段先保持兼容；等前端、日志、看板迁移后再收窄语义。
- `Deny` 不等于 `RequiresApproval=true`。Deny 表示不可执行或需要用户补参数/换意图，不能在展示层被解释成“去审批即可”。

### 4.3 Action 分类

把当前 `mixedActionRules` 拆出更细粒度：

```go
type mixedActionRule struct {
	Field                   string
	ReadOnlyActions         []string
	RoutineSideEffectActions []string
	PrivilegedActions       []string
	DestructiveActions      []string
}
```

`feishu_api` 期望分类：

- `ReadOnlyActions`: `search_contacts`, `get_user_info`, `get_doc_content`, `read_sheet`, `list_tasks`, `download_message_resource` 等只读动作。
- `RoutineSideEffectActions`: `send_message`。
- `PrivilegedActions`: `send_image`, `send_file`, `upload_image`, `upload_file`, `create_task`, `complete_task`, `write_sheet`, `create_bitable_record`, `update_bitable_record`, `create_approval`。
- `DestructiveActions`: 未来有删除类飞书 action 时放这里。

`im_api` / `send_im_message` 期望分类：

- `search_recipients`, `list_recent_conversations`, `resolve_recipient`: read-only。
- `send_message`: routine side effect，前提是平台匹配、recipient/conversation 合法、content 非空。
- 文件、图片、富文本群发等未来 action 默认 privileged。

### 4.4 Routine Send 边界

`send_message` 不是天然安全动作。它只有在以下条件全部满足时才是 `routine_side_effect`：

- `msg_type` 为空或明确等于 `text`。
- `content` / `text` / `message` 非空，且长度小于实现中定义的保守上限。
- recipient / receive_id / conversation_id 已解析，不能缺失。
- 不包含 `mention_all`、`@all`、群发、模板卡片、互动卡片、附件、图片、文件、音频、视频、media key。
- 平台能力匹配当前 intent，例如飞书 intent 才能路由到 `feishu_api`。

任何 card/template/media/file/mention_all/批量外发都必须是 `privileged_side_effect` 或更高风险。缺 content/recipient 是 invalid input，不是 ask/HITL。

## 5. Phase 计划

这次改造不能用一个大 PR 一次性压上生产。分三段，前一段必须可独立上线和回滚。

| Phase | 目标 | 范围 | 上线条件 |
