# media 工具域边界评估

**Status:** COMPLETED

**结论：** v1 不合并 image / video / audio / speech 工具。保留按媒体类型拆分的工具入口，只统一路由分类、prompt 选择口径和验收观察。

## 评估目标

本评估用于关闭“工具域统一与分层收敛总路线”的 Phase 5。它不新增 `media.action`，不迁移 `generate_image`、`generate_video`、`text_to_speech` 等工具。

## 现有能力

- `generate_image`：图片生成，输入多为 prompt、尺寸、风格和输出资产。
- `generate_video`：视频生成，输入包含时长、镜头、素材或 prompt。
- `text_to_speech`：语音合成，输入包含文本、音色、格式。

这些能力共享“媒体生成”大域，但底层模型、参数结构、成本、输出资产和失败模式不同。

## 不合并理由

1. 参数结构差异大。

图片、视频、语音的参数没有稳定交集。强行合并为 `media.action` 会产生大量 action 专属字段，schema 可读性下降，也会提高模型构造错误参数的概率。

2. 成本和执行风险不同。

视频生成通常成本和耗时更高，语音可能涉及声音授权，图片更关注尺寸和资产交付。统一工具名不能降低风险，反而会让审批和配额策略难以按媒体类型表达。

3. 当前没有重复执行路径。

Phase 1 filesystem 和 Phase 2 IM 的核心问题是多个入口做同类执行；media 域目前不是这种重复实现问题。按“复用现有能力、避免重复造轮子”的原则，不应为了形式统一制造新抽象。

## 后续口径

- 不新增 `media.action`。
- 保留媒体类型专用工具。
- 路由层统一归到 `media` domain 即可；需要更细粒度治理时按工具名或 action 专项补策略。
- prompt 中只需说明不同媒体请求选择对应工具，不需要引导模型使用统一入口。
