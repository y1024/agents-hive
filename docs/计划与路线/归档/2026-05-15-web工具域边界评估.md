# web 工具域边界评估

**Status:** COMPLETED

**结论：** v1 不合并 `websearch` / `web_search`、`webfetch` / `web_fetch`、`browser_interact`。保留现有工具边界，只在路由、prompt 和验收口径上收敛。

## 评估目标

本评估用于关闭“工具域统一与分层收敛总路线”的 Phase 4。它不是实施计划，不新增 `web_api.action`，也不迁移现有 web 工具。

## 现有能力

- `websearch` / `web_search`：搜索型只读工具，适合发现外部资料。
- `webfetch` / `web_fetch`：单页抓取型只读工具，适合读取已知 URL。
- `browser_interact`：浏览器交互工具，可能产生登录、点击、表单、截图等状态变化。

这些工具虽然都属于 web 域，但风险模型不同：搜索和抓取是只读外部信息获取，浏览器交互是状态型 UI 自动化，可能触发外部副作用。

## 不合并理由

1. 风险语义不一致。

`websearch` / `webfetch` 可以按 read-only 处理；`browser_interact` 需要 command/action 级约束。把三者合并到单一 `web.action` 会让 read-only 与交互副作用混在一个工具名下，增加 route input gate 和 prompt 解释成本。

2. 现有 mixed action 基础设施已经能覆盖需要的动作约束。

如果后续要治理 `browser_interact.commands[].action`，应复用 `mixedActionRules -> AllowedToolInputs -> runtime input gate` 链路，而不是新建 web 专用策略系统。

3. 迁移收益不足以覆盖兼容成本。

web 工具当前命名重复主要来自历史别名和外部 MCP 风格差异。直接合并会影响工具召回、已有 journal/replay、前端展示和回归语料；收益低于 filesystem/IM 这类强重复执行路径。

## 后续口径

- 保留 `websearch` / `web_search`、`webfetch` / `web_fetch` 兼容。
- 不新增 `web_api` 或 `web.action`。
- 如需收敛，优先做 prompt 优先级、路由语料和别名文档，不做执行层迁移。
- `browser_interact` 若要治理，只在现有 mixed action 链路中补 action 约束和测试。
