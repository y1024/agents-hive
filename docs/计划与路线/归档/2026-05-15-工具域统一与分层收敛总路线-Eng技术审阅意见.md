## TL;DR

**总路线的方向是正确的**:按"能力域 + action 字段"分层收敛,不强求所有工具改名成一个单工具,这是健康的判断,跟 2026-05-14 已落地的统一策略架构是一脉相承的。

**但本计划有 4 个结构性问题必须在动 Phase 1 之前处理:**

1. **Task 1(工具域统一基础设施)与 filesystem 子计划 Task 3 严重重叠**。两边都列出 `capability_registry.go / tool_policy.go / decision.go / toolruntime/action.go / tool_visibility.go`。读者不知道谁是 source of truth,实施时会出现"两个 PR 都改同一份 mixedActionRules"或者"两边都假设对方做了"的事故。
2. **Task 1 描述的 `mixedActionRule` 五字段结构 99% 已经在线**。`internal/router/capability_registry.go:154-161` 一字不差就是这个结构,且 5 个目标工具(feishu_api/im_api/memory/taskboard/browser_interact)都已经注册并被 `tool_policy.go:138-197` 的 `evaluateMixedToolPolicy` 完整消费。Task 1 应该改为"**把 filesystem 加进既有 mixedActionRules**",而不是"建立"。
3. **`send_im_message` "降级为兼容壳"语义未定义**。当前 `internal/tools/sendim.go:73-122` 是 139 行独立工具,直接调 `imcore.SendTarget`,**根本不是** `im_api.send_message` 的封装。计划没说"兼容壳"具体是什么形态(删除/forward/仅改 description),这是真正的设计空白。
4. **第 5 节列的可观测指标有一半在仓库里不存在**(`tool_call_total`、`route_input_denied_total`、`tool_search_recall_total`、`tool_visibility_trimmed_total`)。实际命名规范是 `hive_*_total`,且统一策略指标已经由 2026-05-14 计划落地为 `policy_decision_total` / `action_guard_ask_total`(见 `internal/agentquality/types.go:20-21`)。运维拿这份计划去 Grafana 配看板会全部对不上。

### 架构发现

**[A1] (confidence 9/10) 总路线 Task 1 与 filesystem 子计划 Task 3 职责重叠,无单一来源真相**
- 证据:
  - 总路线 Task 1 Files 列表(行 211-220):`capability_registry.go / tool_policy.go / decision.go / toolruntime/action.go / tool_visibility.go` 加 5 个测试
  - filesystem 子计划 Task 3 Files 列表:同样的 5 个文件 + 5 个测试
- 风险:实施时会出现"Task 1 PR 与 Task 2 PR 都改 mixedActionRules,后合并的覆盖前一个;或者两边都假设对方做了"。
- **建议**:总路线 Task 1 改成"**Phase 0 复用清单**",显式声明这些设施已存在;真正的新增条目移到 filesystem 子计划。Task 1 本节只留:
  - "filesystem 进入 mixedActionRules"作为 cross-reference 子计划
  - 其它工具(im_api/memory/taskboard/browser_interact)标记 ✅ 已落地

**[A2] (confidence 9/10) `IsMixedReadWriteTool` 是 filesystem 安全模型的隐藏硬依赖,未在总路线标记**
- 证据:
  - `internal/router/capability_registry.go:372` `IsMixedReadWriteTool` 的判定就是 "mixedActionRules 是否含此工具"
  - `internal/master/tool_visibility.go:597-608` `defaultRuntimeAllowedInputsForTool` 的入口是 `if callable && router.IsMixedReadWriteTool(name)` — 只有这个分支命中,session 才会自动塞 `{action: "list|glob|grep|read"}` 默认值
  - `internal/router/tool_policy.go:101-103` 也是 `if IsMixedReadWriteTool(profile.Name) { return evaluateMixedToolPolicy(...) }`
- 风险:**如果 filesystem 工具先注册了、但 mixedActionRules 没加 filesystem 条目**,plan mode 下 `filesystem.action="edit"` 会因为 session 无 allowed inputs 而被 `RouteInputDenyReason` 放行 — 计划模式被绕开。
- 修复:filesystem 子计划 Task 3 Step 3.2 已经写了这一步,但总路线没把它标成 Phase 1 的 P0 硬依赖。
- **建议**:总路线 §4 Phase 1 显式加一条"Phase 1 必须把 filesystem 加入 `mixedActionRules`(不能延后到 Phase 2)",并在 §6 回滚策略里要求"回滚 filesystem 时同时回滚 mixedActionRules 条目,否则 IsMixedReadWriteTool 会返回 true 但 evaluateMixedToolPolicy 找不到 action"。

**[A3] (confidence 8/10) `multiedit` / `multi_edit` 命名分裂未处理**
- 证据:
  - `capability_registry.go:95-96` 双注册 `multi_edit` + `multiedit`
  - `capability_registry.go:265` `hostToolGroups["fs"]` 同时含两种写法
  - `internal/master/react_processor.go:2094,2195` 双 case
  - 实际工具 `internal/tools/multiedit.go:61` `Name: "multiedit"`(无下划线)
  - **filesystem 子计划 §4 schema enum 用了 `"multi_edit"`(带下划线)**
- 风险:
  - 短期:filesystem.action="multi_edit" 调用是 OK 的(它只走 filesystem executor,不查 builtinToolRules);
  - 中期 Phase 7 "降级旧入口"时,如果操作把 `multiedit` 从 `hostToolGroups["fs"]` 删了,所有现存 `multiedit` 直接调用立刻断;反之删 `multi_edit` 也会破坏 `react_processor.go:2094` 双 case 之一。
- **建议**:Phase 1 之前把命名统一成一个,推荐保留 `multiedit`(单 token,与 Anthropic 内置工具同名,降低 LLM token 成本)。在子计划 §4 schema enum 改成 `"multiedit"`,代码侧把 builtinToolRules/hostToolGroups/react_processor 三处的 `multi_edit` 形式删掉。

**[A4] (confidence 7/10) Phase 2(IM 收敛)缺迁移窗口设计**
- 证据:filesystem 子计划有 Phase 1 (7 天)/ Phase 2 (14 天)/ Phase 3 决定的指标驱动迁移;Phase 2 IM 收敛只有 Step 3.1-3.4 工程步骤,没有"观察 N 天 + 哪个指标作为下线触发"。
- 风险:`send_im_message` 仍被 skill / sub-agent / 默认 prompt 大量引用(`hostToolPolicyProfiles["messaging"]={"send_im_message", "feishu_api", "skill"}`,`capability_registry.go:277`),贸然降级会让现存集成失败。
- **建议**:为 Phase 2 引入与 filesystem 同结构的三阶段迁移,并明确"send_im_message 调用量下降到 X% 以下 + im_api.send_message 错误率 < Y%"作为 Phase 3 触发条件。

**[A5] (confidence 8/10) `send_im_message` "降级成兼容壳"是真正未决的架构决策**
- 证据:`internal/tools/sendim.go:73-122` 当前是 139 行独立工具,直接 `service.SendMessage(ctx, imcore.SendTarget{...})`,不依赖 `im_api` 任何代码。
- 三种可能:
  - **形态 A — 删除**:Phase 3 直接移除 sendim.go,顶层只剩 im_api。优点:DRY;缺点:违反"先迁移后下线"原则,会立刻打断现存调用。
  - **形态 B — 内部 forward**:sendim.go handler 把 `{platform, chat_id, content}` 转成 `im_api.send_message {action,platform,recipient_id,content}` 并复用 im_api 的 executor + audit。优点:单一执行路径,审计 100% 一致;缺点:需要在 im_api.go 抽出 `executeIMAPISendMessage(ctx, params)` 供 sendim.go 调用,且要重映射 schema(`chat_id`→`recipient_id`)。
  - **形态 C — 仅改 description**:行为不变,只改文案降低模型选择优先级。优点:零代码改动;缺点:不解决"两套审计/两套错误分支"的问题,且 `policy_consistency_test.go:44-52` 已经把 send_im_message 当 routine_side_effect 验证,语义不会因 description 变。
- **强烈建议**:形态 B。理由:
  - 当前 `policy_consistency_test.go` 已经在断言 send_im_message 和 im_api.send_message 是同一 risk class,如果两套 executor 行为漂移,这个一致性测试只能验路由不能验执行
  - 形态 B 让"兼容壳"有可观测语义:`im_send_via_legacy_total` 指标可以追踪并触发 Phase 3 下线
- **决策点(需用户拍板)**:形态 A / B / C,作为 Phase 2 的设计前提

**[A6] (confidence 7/10) 失败模式覆盖不足**
对于本计划新增的代码路径,生产失败场景遗漏:

| 新代码路径 | 实际失败场景 | 计划是否覆盖 |
|---|---|---|
| filesystem.action 输入 schema 校验 | 模型传错字段(如 grep 不带 pattern) | 子计划 Step 2.3 已覆盖 ✅ |
| filesystem 与旧工具 read tracker 共享 | filesystem.read 后旧 edit 失败 | 子计划 Task 2 Step 2.5 已覆盖 ✅ |
| im_api 调用时 platform 字段缺失 | recipient_id 平台无法识别 | **未覆盖** ❌ |
| plan mode 下 filesystem 进入但 mixedActionRules 没加 | action=edit 绕过 gate | **未覆盖**(见 A2) |

### Critical 级(协议层 + 生命周期)

#### [F1] (confidence 10/10) **`IsConcurrencySafe` 是工具级 boolean,filesystem 无法表达 per-action 并发安全性 — 性能回归 Critical**

证据:
- `internal/mcphost/host.go:23` `IsConcurrencySafe bool` 是 ToolDefinition 字段,**工具级**
- `internal/tools/batch.go:171` `if !def.IsConcurrencySafe { ... }` — batch 用这个 flag 决定能否并发
- `internal/tools/tools.go:413,670,743` read_file/glob/grep 都设 `IsConcurrencySafe: true`
- write_file/edit/multiedit 隐式 false

filesystem 是**单一**工具:
- 设 `true` → batch 内并发 filesystem.write/edit/multi_edit → 文件锁竞争、数据破坏
- 设 `false` → batch 内 filesystem.list/glob/grep/read **无法并发** → batch 工具的核心性能价值消失

**性能影响量化**:模型常用 batch 同时 grep 5 个 pattern 或并发 read 10 个文件,这种场景从 ~200ms 退化到 ~1s+(串行)。

**建议**(优先级排序):
- A. 子计划 Task 1 抽 executor 时,**保留旧 read_file/glob/grep 注册**(IsConcurrencySafe=true),让模型 batch 时仍走旧入口;Phase 7 不下线只读旧入口
- B. 扩展 mcphost 协议:`IsConcurrencySafe` 改成 `ConcurrencySafeActions []string`,batch 工具按 action 判断 — 改协议成本高
- C. 在 batch 工具内 special-case filesystem,按 action 决定并发 — 把 batch 与 filesystem 强耦合,不优雅

推荐 A,因为它与子计划"旧工具保留兼容"原则一致,且不破坏现有 batch 性能。**但子计划 Phase 7 必须明确"只读旧工具不下线",或者协议层升级。**

#### [F2] (confidence 9/10) **`RegisterBuiltinTools` 只在 startup 调用一次,Feature flag 无法 hot apply — [S6] 假设错误**

证据:
- `cmd/.../bootstrap/server.go:486` `tools.RegisterBuiltinTools(...)` 是唯一调用点
- 没有 `ReloadTools` / `HotReload` 等运行时重注册接口
- `mcphost.Host.RegisterTool` 是 register-once 语义

修正第二轮 [S6] 建议:`config.Tools.FilesystemEnabled` 在启动时检查 OK,**但运行中改配置无法生效**。生产事故时,值班同学修了 config 后**仍需重启服务**才能关闭 filesystem。

**真正的 feature flag 实现**:
- 不是在 `registerFilesystem` 里 `if !enabled return`(那只在 startup 检查)
- 而是在 `tool_visibility.go` 默认可见集合 + `tool_policy.go` evaluate 路径上,运行时读 config:
  ```go
  if !cfg.Tools.FilesystemEnabled {
      // 把 filesystem 从可见集合排除,evaluate 直接 deny
  }
  ```
- 或者维护一个 `runtimeDisabledTools sync.Map`,通过 admin API 动态写入

**建议**:子计划 Task 4 加 step "filesystem 在 tool_visibility 层支持 runtime disable,通过 `config.update` API 立即生效"。

#### [F3] (confidence 9/10) **错误信息硬编码中文 + 直接拼接 err.Error() — i18n 缺位 + 路径泄露 Critical**

证据:`internal/tools/tools.go:430,442,448,467,474` 全部:
```go
return errorResult("读取失败: " + err.Error()), nil
return errorResult(fmt.Sprintf("文件过大 (%d 字节), 超过 %dMB 限制", ...)), nil
```

风险:

- 中文写死,英文用户看到中文报错

**建议**:filesystem 是统一机会,Phase 1 加错误归一:

这是子计划完全没碰的现代化机会。
