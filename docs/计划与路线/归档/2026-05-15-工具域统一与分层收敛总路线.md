**Goal:** 把分散工具入口按能力域收敛成少量结构化工具族，先完成 `filesystem`，再把纯 IM 消息发送收敛到 `im_api`，同时保持 `memory`、`taskboard`、`browser_interact`、`bash`、planning、customer_service 等既有安全边界稳定。

**Architecture:** 复用 2026-05-14 已落地的统一策略链路：`builtinToolRules` / `mixedActionRules` -> `InferToolProfile` / `ToolActionProfile` -> `EvaluateToolPolicy` -> `RouteDecision.AllowedToolInputs` -> runtime input gate。新增工作不是重建 mixed action 基础设施，而是把 `filesystem` 接入既有 mixed action 规则，明确 `send_im_message` 作为 `im_api.send_message` 的兼容转发壳，补齐 DB 默认权限、WebUI 文案、agentquality 语料、prompt 优先级、feature flag 和迁移指标。

**Tech Stack:** Go backend (`internal/tools`, `internal/router`, `internal/master`, `internal/toolruntime`, `internal/store`, `internal/config`, `internal/bootstrap`), React i18n (`frontend/src/i18n/locales`), agentquality corpus, table-driven Go tests.

### 1.1 实施细节调整：复用是硬门

实施口径从“新增统一工具族”调整为“在已有能力上增加统一入口”。后续实现必须先证明复用关系，再写代码。

具体调整：

- `filesystem` 不是新文件系统 runtime。它只能是 `action` dispatcher：参数强校验、参数映射、调用旧工具 executor / helper、错误脱敏、迁移指标。
- 旧文件工具仍是 source-of-truth。`read_file`、`write_file`、`glob`、`grep`、`ls`、`edit`、`multiedit` 的路径处理、输出格式、锁、read tracker、file tracker、diagnostics 不得被复制一份。
- Task 1 不再理解成“为了新工具大重构所有旧工具”。正确做法是先交 source-of-truth 映射表；只有旧 handler 闭包无法共享时，才抽最小 private helper。
- `send_im_message` 不再保留独立发送实现。它只做 legacy 参数到 `im_api.send_message` 的 mapper，并共享 `imcore.Service.SendMessage`。
- 策略层、Plan mode、runtime input gate、ActionGuard 继续使用 2026-05-14 已完成链路。不得新增平行 policy package 或绕过既有 gate。
- metrics / audit / replay / journal 都优先扩展现有链路；除非现有链路无法表达迁移看板，才新增低基数字段指标。

这意味着代码 diff 应该主要表现为：

- mapper、thin wrapper、少量 helper extraction。
- `mixedActionRules` / `builtinToolRules` / visibility set / allowed inputs 的增量配置。
- 既有测试的回归锁定和 migration fixture 增量。

如果实现中出现大段新文件搜索、文件编辑、权限判断、IM 发送或回放系统代码，默认视为偏离计划，需要先退回设计审查。

## 4. Phase 1: filesystem

详细实施以子计划为准：

- [2026-05-15-filesystem统一工具实施计划.md](/Users/guoss/workspace/company/vast/agents-hive/docs/计划与路线/归档/2026-05-15-filesystem统一工具实施计划.md)

总路线只声明 Phase 1 的硬门：

- `filesystem` 必须加入 `mixedActionRules`，否则 plan mode 只看工具名会存在 action 写入绕过风险。
- `filesystem` 必须受 `RouteDecision.AllowedToolInputs["filesystem"]["action"]` 约束。
- `filesystem` 必须有 feature flag：`config.Tools.FilesystemEnabled`。
- `filesystem` 必须进入 DB 默认权限 JSON。
- `filesystem` 必须有 WebUI i18n 名称和 action 文案。
- `filesystem` 必须更新 agentquality route corpus。
- `filesystem` 必须明确与 `apply_patch` 的工具选择优先级。
- `filesystem` v1 必须保持工具级 `IsConcurrencySafe=false`，因为调度器只按工具名判断并发安全，不能按 action 判断。
- 旧 `ls` / `glob` / `grep` / `read_file` 必须长期保留为并发安全的只读直连工具；除非先升级 `mcphost` 支持 per-action concurrency，否则不得把这些只读旧工具彻底下线。
- `filesystem` 的 action 安全边界必须由 router / runtime gate / ActionGuard 承担，不能依赖 HITL `PermissionRule.pattern`。
- `filesystem.write/edit/multiedit` 必须复用旧写入路径的文件锁、read-before-write/edit、FileTracker 和 LSP diagnostics。
- `filesystem.go` 必须保持 thin dispatcher；不得包含独立目录遍历、grep、文件读写、字符串替换、多编辑回滚或诊断调度实现。
- `filesystem.edit` 在 Plan mode / read intent 下必须被 route input gate 拒绝，且拒绝要有 ToolCallEvent、quality event 和 `hive_route_input_denied_total`。
- feature flag 不能只在 `RegisterBuiltinTools` 启动注册点生效；还必须在 tool visibility / policy / route admission 层支持运行期禁用语义，否则配置更新后仍会继续向模型暴露 `filesystem`。
- fast-path trimming 的优先级表必须把 `filesystem` 放在旧只读文件工具之前，否则 `filesystem` 会被 `read_file|grep|glob|ls` 挤出模型可见列表。
- `filesystem` 必须有 action 级 audit，字段只能记录 `action`、`path_hash` / basename、bytes、status、duration 等低敏信息，不能记录完整 path、content、old_string、new_string。
- 前端 replay timeline 和 journal 必须把 mixed action 工具显示为 `tool.action`，例如 `filesystem.grep`、`filesystem.edit`、`im_api.send_message`。
- `logFileChangeIfNeeded` / `changedFilesFromToolCall` 必须识别 `filesystem.write/edit/multiedit`，否则 replay 和审计会漏掉文件变化事件。
- `ToolResultBudgetCompactor` 与 `ReadTracker` 的耦合必须有明确测试：read 结果被压缩或服务重启后，写入应要求模型重新读取，不能只靠旧的 path 标记放行。
- agentquality 不只迁移 `route_eval/corpus.json`，还必须迁移 `aq*.json` 和 `longrun/` fixture，并 dry-run `route-eval` / `memory-nightly-eval` 相关 workflow。
- 权威文档 `docs/架构设计/Tool-Routing.md`、`docs/架构设计/安全权限模型.md` 需要同步 `filesystem` 的 action 风险和 plan mode 行为。

## 12. 回滚策略

### 12.1 filesystem

必须通过配置开关回滚：

```json
{
  "tools": {
    "filesystem_enabled": false
  }
}
```

实现要求：

- `RegisterBuiltinTools` 中 `cfg.Tools.FilesystemEnabled == false` 时不注册 `filesystem`。
- `config.Tools.FilesystemEnabled` 使用 `*bool` 或等价 tri-state 表达，避免配置省略和显式 `false` 混淆。
- 默认值为 true，并通过 `cfg.Tools.IsFilesystemEnabled()` 判断。
- 配置关闭不影响旧工具。

### 12.2 IM

回滚方式：

- 保留 `send_im_message` 工具名。
- 如果 forward 出现事故，开关切回旧 send executor。
- 如果没有开关，则必须保留一个小型 fallback function，不直接删除旧发送逻辑。

### 12.3 数据与 UI

回滚 filesystem 时：

- 不需要 DB 数据迁移回滚。
- 允许 i18n 保留 `filesystem` 文案。
- `pgDefaultPermissionRulesJSON` 可以保留 `filesystem` 规则，因为工具未注册时规则无效。
