# CRUD/DTO/Runtime Boundary Refactor Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Separate display DTOs, write DTOs, persisted records, runtime state, and secret-bearing configuration so browser/cache state cannot overwrite backend truth.

**Architecture:** Treat this as a boundary refactor, not a page cleanup. Backend request types become the source of write semantics; frontend client methods stop accepting broad display records; store methods get explicit create/update/runtime names where upsert currently hides intent. Existing security merge helpers stay as compatibility guards, but new code should not rely on masked view objects being written back.

**Tech Stack:** Go HTTP/RPC handlers, Go store interfaces, React/TypeScript admin UI, Vite, Vitest, Go tests.

---

## Evidence Summary

This is based on repository scans run on 2026-05-16:

- 1157 Go/TS/TSX files under `internal/` and `frontend/src/`.
- 34 frontend matches for high-risk write entrypoints such as `updateRuntimeConfig`, `saveExternalResource`, scheduled task upserts, auth provider admin writes, and LLM provider/model admin writes.
- 23 frontend matches using broad `Partial<Record|Config|Task|Resource|Provider|Model>` patterns. Some are harmless form-initial-state usage; the risky ones are public client method signatures and save payloads.
- 61 store interface methods in `internal/store/types.go`; at least 10 are `Save*` methods with upsert-like semantics or broad record parameters.
- 17 duplicate or near-duplicate helper patterns around clone, normalize, redact, merge, pagination, and submit payload construction.
- 28 `Record` types/interfaces across backend/frontend, several of which are used as both persisted/view shapes and write shapes.

The confirmed issue count is not 34. Current confirmed actionable issues are 8 primary issues plus 4 duplicate-helper cleanup targets. The rest are audit candidates to verify and either classify as safe or convert into explicit DTOs.

## Confirmed Issues

### P0/P1 Boundary Issues

1. **Frontend admin client still exposes broad write signatures.**
   - `frontend/src/api/node-client.ts`
   - `adminCreateProvider`, `adminUpdateProvider`, `adminCreateLLMProvider`, `adminUpdateLLMProvider`, `adminCreateLLMModel`, `adminUpdateLLMModel`, `saveExternalResource`
   - Risk: callers can pass display records or server records back into write endpoints.

2. **Runtime config writes remain one large mixed endpoint.**
   - `internal/gateway/methods_config.go`
   - `frontend/src/components/settings/*`
   - Current secret merge guards are good, but `config.update` still mixes agent, channel, MCP, HITL/security, and permission concerns in one request shape.

3. **Scheduled task definition and runtime state are mixed in the same API type.**
   - `internal/store/types.go`
   - `frontend/src/types/api.ts`
   - `internal/api/scheduled_tasks_handler.go`
   - Fields such as `last_run_at`, `next_run_at`, `active_run_id`, and `lease_expires_at` are exposed beside definition fields. Store SQL protects part of this, but the type boundary is still wrong.

4. **Store `Save*` names hide create/update/upsert semantics.**
   - `SaveScheduledTask`, `SaveMCPServer`, `SaveExternalResource`, `SaveLLMProvider`, `SaveLLMModel`, `SaveChannelConfig`, etc.
   - Risk: future handlers may treat a partial object as a full record.

5. **LLM admin page has touched-only logic, but client types do not enforce it.**
   - `frontend/src/pages/admin/LLMProviders.tsx`
   - `frontend/src/api/node-client.ts`
   - The UI now builds touched payloads, but the API client still accepts `Partial<LLMProviderRecord>` and `Partial<LLMModelRecord>`.

6. **Auth provider admin write types are still broad on the frontend.**
   - `frontend/src/api/node-client.ts`
   - `frontend/src/pages/admin/AuthProviders.tsx`
   - Backend redaction is now centralized, but frontend write contracts still do not express create vs patch.

7. **External resource save still looks like upsert from the frontend.**
   - `frontend/src/components/settings/ExternalResourcesSettings.tsx`
   - `frontend/src/api/node-client.ts`
   - Backend preserves credentials on omitted fields now, but client types should encode omitted vs empty semantics.

8. **`config.update` sub-DTOs use mixed pointer and non-pointer semantics.**
   - `internal/gateway/methods_config.go`
   - MCP server update uses pointer fields, but channel structs still use full config structs and compatibility merge helpers.

### Duplicate/Near-Duplicate Helper Issues

9. **Map clone helpers are duplicated.**
   - `internal/gateway/methods_config.go:642`
   - `internal/master/tool_visibility.go:824`

    - `internal/security/redaction.go` is the correct common layer.
    - `internal/api/admin_handlers.go:37` and `internal/api/llm_handlers.go` still expose local wrappers for historical reasons.

## Non-Goals

- Do not redesign all admin pages.
- Do not remove existing endpoints immediately.
- Do not change database schema unless a task explicitly calls for it.
- Do not rewrite the Store interface all at once.
- Do not weaken the existing secret merge guards; they remain compatibility protection.

## Target Architecture

Each configurable object gets five shapes where needed:

- `Record`: database shape, may include secrets and runtime fields.
- `View`: response shape, redacted and safe for browser state.
- `CreateRequest`: full write shape, only fields valid on creation.
- `UpdateRequest`: patch shape with pointer/optional fields; omitted means unchanged, explicit empty means clear.
- `RuntimeState`: fields mutated by schedulers, claims, reloaders, or execution loops.

New or changed frontend client methods must accept create/update request types, not `Partial<Record>`.

func CloneStringMap(in map[string]string) map[string]string {
	if in == nil {
		return nil
	}
	out := make(map[string]string, len(in))
	for k, v := range in {
		out[k] = v
	}
	return out
}

func CloneNestedStringMap(in map[string]map[string]string) map[string]map[string]string {
	if in == nil {
		return nil
	}
	out := make(map[string]map[string]string, len(in))
	for k, v := range in {
		out[k] = CloneStringMap(v)
	}
	return out
}

func parseBoundedPagination(r *http.Request, defaultSize, maxSize int) (page, size int) {

	size, _ = strconv.Atoi(r.URL.Query().Get("size"))

2. Split frontend write DTOs.
3. Fix LLM/Auth/ExternalResource client contracts.
4. Split runtime config patch semantics.
5. Split scheduled task definition/runtime state.
6. Add store narrow methods and migrate callers.
7. Consolidate duplicate helpers.
8. Enable CI guard.

## Residual Risk

- This plan intentionally leaves compatibility wrappers in place at first. A second cleanup pass should remove deprecated `Save*` entrypoints after all callers migrate.
- `config.update` is kept for compatibility. New section-specific RPC methods can be introduced later if the UI needs clearer network boundaries.
- Some duplicate helpers should remain local if their semantics differ. Do not centralize based only on similar names.

## Completion Criteria

This work is complete only when:

- Frontend write APIs no longer accept `Partial<Record>` shapes for sensitive resources.
- Backend write handlers use explicit create/update DTOs.
- Runtime state cannot be overwritten through definition update paths.
- Store write methods communicate create/update/upsert intent.
- Duplicate helpers have either been consolidated or deliberately documented as semantically distinct.
- Full Go tests, frontend lint/build/tests, and static boundary guard pass.
