### Markdown 图片归档规则

P2 对 Markdown 图片采用“对象体、元数据、文本引用三分离”：

- 图片对象体：写入统一对象存储。local provider 写到 `asset.local.base_path`，MinIO/S3 provider 写到配置 bucket，例如 `hive-assets`。
- 资产元数据：写入 `assets` 表，包含 `owner_scope/owner_id`、asset namespace、content hash、mime、size、filename、object key 和 tags。
- KB 关系：新增 `kb_node_assets` 或等价字段，记录 `namespace_id/document_id/node_id/asset_uri/alt_text/caption/content_hash`。
- Markdown/tree node text：只保留 `asset://...`，不保留 base64、不保留本地相对路径、不保存二进制。

输入处理规则：

- `![alt](./images/a.png)`：从上传包或同批 multipart 文件中取图，上传 asset，替换为 `![alt](asset://kb/{owner_scope}/{owner_id}/{namespace_id}/{document_id}/{content_hash}.png)`。
- `![alt](data:image/png;base64,...)`：解码后上传 asset，Markdown 中替换为 `asset://`；超大小限制直接拒绝。
- `![alt](https://...)`：P2 首发不默认抓取公网图片，除非管理员显式开启 allowlist 下载；默认要求重新上传图片包并拒绝 ingest，不生成 active document。
- 引用缺失的本地图片：ingest fail，不得生成 active document。
- 相同 owner + asset namespace + content hash 去重；不同 owner 不能共享 metadata 授权，即使 object key 去重也必须 owner-scoped。

上传入口规则：

- Admin KB 导入页和 `NodeClient.ingestKBDocument` 只使用 `FormData` / `multipart/form-data`，文档文件字段为 `file`，图片字段为重复 `assets`，文本 Markdown 字段为 `markdown` 或 `content`。
- 服务端直接把 multipart 上传字节交给 `MarkdownRegistry.Convert`，不把文档文件重编码为 JSON/base64。系统未上线，无老客户端兼容负担；JSON/base64 ingest API 已删除，非 multipart 请求返回 415。
- multipart 图片默认用上传文件名匹配 Markdown 引用；需要保留目录时可传同序 `asset_path`、`asset_alt_text`、`asset_caption` 字段。
- 已存在的 `asset://` 图片引用可走普通 Markdown ingest；本地路径、data URI、远程 URL 等非 `asset://` 图片引用都会进入 asset ingest，其中远程 URL 默认拒绝。

展示和取证规则：

- `asset://` 不是公开 URL。前端必须调用 `/api/v1/assets/resolve?uri=...`，服务端校验 owner/domain/binding/document 权限后返回短时 signed URL 或代理 URL。
- `kb.section.text` 返回 `asset_refs`，不要把图片 base64 放入 tool output。
- 图片内容不能自动成为文本证据。没有 OCR/vision extracted text 时，citation 只能证明“节点包含该图片”，不能证明图片中的事实。

- [ ] **Step 2: PDF Provider 抽象与默认 MinerU**

P2 首发策略：

- PDF 不走 stub，不做“文本抽取受限降级”。PDF 转 Markdown 必须由 `MarkdownProvider` 完成。
- 默认 provider 为 MinerU：`fileconv.markdown.pdf.provider=mineru`，默认命令形态为 `mineru -p {input} -o {output}`。由于 MinerU CLI 版本可能变化，必须允许通过 `fileconv.markdown.pdf.command.args` 或环境变量覆盖。
- 配置选择 MinerU 时，server 启动期必须检查 MinerU binary 是否存在；不存在且 `fileconv.markdown.pdf.install.enabled=true` 时执行配置的安装命令，安装失败则服务 fail-fast。默认安装命令为 `builtin:python-venv-pip`，即在 `install_dir` 创建隔离 Python venv（优先 `python -m venv`，失败后 fallback 到 `python -m virtualenv`）并安装 `mineru[all]`，再把运行 binary 指向该 venv 下的 `mineru`。不得等到用户上传 PDF 时才静默发现缺依赖。
- Docker runtime 镜像应直接安装 Python、uv 和 `mineru[all]`；启动期自检/自动安装只作为本地部署或镜像漂移的兜底。MinerU 官方 CLI 形态为 `mineru -p <input_path> -o <output_path>`，CPU/无 GPU 部署可通过配置覆盖 args 增加 `-b pipeline`。
- 允许 `provider=external` 接其他 OCR、解析工具或模型服务，只要输出 Markdown 文件和图片资产目录即可复用同一 ingest 链路。
- 允许 `provider=none` 显式关闭 PDF ingest；关闭或 provider binary 缺失时，API 返回 503，不生成 active document。
- Provider 必须把 Markdown 中引用的图片资产以 `ExtractedAsset.Path` 返回，后续统一交给 `internal/kb/asset_ingest.go` 上传到 `internal/asset` 并重写为 `asset://`。
- Provider 必须校验 PDF header、超时、空 Markdown、命令失败和输出缺失；失败可观测，不能吞错。
- `MarkdownDocument.Provider` 记录实际 provider 名称，供 API、日志和后续 QualityWorkbench 展示。
- 不使用 PageIndex 的 PDF LLM fallback。
- PageIndex 开源仓库没有 MinerU/OCR/图片资产输出实现；只吸收其页级边界校验、严格失败和未来智能 tree 的工程教训。`verify_toc`、页码偏移估计、错误 TOC 修复、大节点递归拆分只作为后续智能 PDF tree 的参考，不进入本任务实现。

- [ ] **Step 3: 接入 AssetService**

KB service 依赖 asset service interface，不直接依赖 MinIO。处理流程：

1. 解析 Markdown image tokens，收集 local/data/remote refs。
2. local refs 从上传包或同批 multipart 文件解析；禁止任意读取服务器文件路径。
3. 调 `AssetService.Upload`，使用 owner-scoped KB asset namespace。
4. Markdown 文本替换为 `asset://` URI。
