## 2. 当前总体判断

### 2.1 方向成立

- 按“能力域 + action 字段”分层收敛是合理方向。
- 不把所有工具强行并入单工具是正确边界。
- 不把 `bash`、agent/delegation、planning、customer_service、web、media 强行合并是正确边界。
- 2026-05-14 工具权限治理已提供策略链路，本计划只应接入和验证，不重建。

### 2.2 真正新增主线

| 主线 | 内容 | 优先级 |
|---|---|---|
| Phase 1 | 新增 `filesystem.action` 统一入口，并迁移文件工具默认选择 | P0 |
| Phase 2 | `send_im_message` forward 到 `im_api.send_message` 共享 executor | P1 |
| Phase 3 | memory/taskboard 仅补回归验证 | P2 |
| Phase 4/5 | web/media 只做边界评估文档 | P2 |
| Phase 6/7 | prompt/visibility 与迁移窗口 | 依赖 Phase 1/2 指标 |

### 2.3 不应重复建设

以下能力已经存在，实施时不得重建：

- `mixedActionRule` 结构。
- `im_api` / `memory` / `taskboard` / `browser_interact` mixed action rules。
- `EvaluateToolPolicy` 和 `evaluateMixedToolPolicy`。
- `MixedAllowedToolInputsForIntent`。
- `RouteInputDenyReason` runtime gate。
- Plan mode 工具名 gate。
- `memory.delete` / `taskboard.delete` 高风险标记。
- agent/delegation 与 `bash` 独立风险画像。

### 2.4 复用硬门

实施细节需要收紧：本计划不是新建一套“工具域平台”，而是把新入口接到已有 source-of-truth 上。任何 PR 缺少复用证据，都不能通过验收。

必须复用：

- 文件执行能力：`internal/tools/tools.go`、`internal/tools/ls.go`、`internal/tools/multiedit.go` 中现有 `read_file` / `write_file` / `glob` / `grep` / `ls` / `edit` / `multiedit` 的输入结构、路径处理、输出格式、错误语义和测试基线。
- 文件状态能力：`globalReadTracker`、`globalFileTracker`、`globalFileLock`、LSP diagnostics 触发路径。不得新建第二套 read-before-edit、文件 hash、锁或诊断系统。
- 策略能力：`builtinToolRules`、`mixedActionRules`、`EvaluateToolPolicy`、`MixedAllowedToolInputsForIntent`、`RouteInputDenyReason`、`EvaluatePlanToolGate`。不得新增平行 policy package、平行 gate 或只在 `filesystem` 内部私判风险。
- 可见性和召回能力：`HostToolSetDefaultVisible`、`HostToolSetPlanAllowed`、`hostToolGroups`、`toolRecallObservation`、fast-path trimming。不得新建第二套工具目录或召回列表。
- 审计和回放能力：`logFileChangeIfNeeded`、`changedFilesFromToolCall`、现有 journal / replay 分类链路。不得绕过现有事件模型另写一份 filesystem timeline。
- IM 发送能力：`im_api`、`imcore.Service`、现有 adapter / router 发送路径。`send_im_message` 只能变成兼容壳，不能保留第三条发送执行路径。
- 指标能力：先 grep `internal/observability` 和现有 metric writer；能扩展现有指标就扩展现有指标，新增指标必须以迁移看板无法表达为理由，且不得带 path、content、session_id、user_id 等敏感或高基数字段。

允许新增的代码形态：

- thin dispatcher：把 `filesystem.action` 映射到旧工具 source-of-truth。
- private helper extraction：从旧工具闭包中抽出最小可复用函数，让旧工具和 `filesystem` 同时调用。
- adapter / mapper：只做参数名映射、action 校验、脱敏错误包装和迁移指标。

禁止新增的代码形态：

- 独立文件搜索引擎、独立读写实现、独立 patch/edit 算法。
- 独立权限策略系统、独立 ActionGuard 替代物、独立 Plan mode gate。
- 独立 IM 发送 service。
- 为了 `filesystem` 改写旧工具行为、输出格式或并发语义。

## 5. Phase 1 filesystem 硬门

Phase 1 必须同时满足以下硬门，缺任意一条都不能算完成：

- [x] `filesystem` 进入 `mixedActionRules`，read actions 为 `list|glob|grep|read`，write actions 为 `write|edit|multiedit`。
- [x] `filesystem` 受 `RouteDecision.AllowedToolInputs["filesystem"]["action"]` 约束。
- [x] read intent 下允许 `filesystem.read/grep/glob/list`，拒绝 `filesystem.write/edit/multiedit`。
- [x] local-write intent 下允许 `filesystem.write/edit/multiedit`。
- [x] Plan mode 下允许 `filesystem.read`，拒绝 `filesystem.edit/multiedit/write`，且拒绝不走 HITL / ActionGuard ask。
- [x] `filesystem.go` 只是 action dispatcher、参数映射、强校验、脱敏错误包装和指标入口，不包含独立文件读写/搜索/编辑算法。
- [x] `filesystem.write/edit/multiedit` 复用旧写入路径的文件锁、read-before-write/edit、FileTracker、LSP diagnostics。
- [x] `filesystem.read/grep/glob/list` 复用旧只读路径的路径解析、输出格式、截断、二进制/PDF/图片处理能力。
- [x] `filesystem` 的 tool definition `IsConcurrencySafe=false`；旧 `read_file/grep/glob/ls` 保持并发安全且长期保留。
- [x] `config.Tools.FilesystemEnabled` 使用可表达显式 false 的配置，并影响注册、visibility、policy/admission。
- [x] `pgDefaultPermissionRulesJSON` 增加 `filesystem`，并停止写回 `multi_edit` 默认规则。
- [x] WebUI i18n 增加 `filesystem` 与 action 文案。
- [x] agentquality 迁移期支持 `filesystem` 与旧文件工具双轨，并标记 preferred/compat。
- [x] replay/journal 展示 mixed action 为 `tool.action`，例如 `filesystem.grep`、`filesystem.edit`。
- [x] file_change 识别 `filesystem.write/edit/multiedit`。
- [x] metrics/audit 不包含 path、content、old_string、new_string、session_id、user_id 这类敏感或高基数字段。
- [x] prompt 明确 `filesystem.edit|multiedit`、`apply_patch`、`bash` 的优先级。

## 7. Phase 3/4/5/6/7 口径

### Phase 3 memory/taskboard

只做回归验证，不重构：

- `memory.search/list` read intent allow。
- `memory.save/update` local-write intent allow。
- `memory.delete` high risk / ask 保持。
- `taskboard.get/list` read intent allow。
- `taskboard.create/update` local-write intent allow。
- `taskboard.delete` high risk / ask 保持。

### Phase 4 web

只写边界评估文档。默认不合并 `websearch`、`webfetch`、`browser_interact`，因为 SSRF、防点击副作用、有状态 browser session 三类风险不应混进同一 action 表。

### Phase 5 media

只写边界评估文档。默认不合并 image/video/audio，因为 schema 差异大，统一 schema 会过宽。

### Phase 6 prompt / visibility

不能作为独立大改直接开工。Phase 1 和 Phase 2 已包含必要 prompt 变更；Phase 6 只做观察后的默认可见性收敛。

### Phase 7 迁移窗口

只能根据指标推进，不允许按代码完成时间推进。旧入口默认可见性降级必须可回滚，旧工具名长期保留。

## 8. 全局验收标准

全局完成必须同时满足：

- `filesystem` 可通过 feature flag 关闭，关闭后旧文件工具仍可用。
- `filesystem` read/write action 在 router、runtime、Plan mode 三层语义一致。
- `filesystem` 写入动作不绕过 read-before-write/edit、FileTracker 和文件锁。
- `filesystem` 的实现证据显示只做 thin dispatcher / mapper / wrapper，没有重复实现文件读写、搜索、编辑、权限 gate 或诊断链路。
- `multiedit` 是唯一新批量编辑 action 名；`multi_edit` 不再作为新 schema / 新策略 / 默认规则出现。
- `send_im_message` 共享 `im_api` send executor 和 `imcore.Service` 发送路径，但仍保留 legacy 工具名。
- DB 默认权限、config 默认权限、WebUI 默认规则一致。
- Replay/Journals/Trace 能显示 mixed action，而不是只显示工具名。
- agentquality route eval 在迁移期能区分 preferred 与 compat。
- 新增 metrics/audit 不泄露路径、内容或高基数字段。
- web/media/customer_service/planning/bash/agent 域边界被文档化，不被误合并。
