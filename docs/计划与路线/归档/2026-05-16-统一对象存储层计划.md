# 统一对象存储层计划（internal/asset）

> **日期**：2026-05-15
> **定位**：Hive 全局文件资产存储基础设施。KB 文档图片、IM 对话附件、Agent 生成物统一走此层。
> **前置**：无。本计划是 `业务域KB与证据层计划总览` 和 `KB P2 资产、PDF/DOCX 与客服试点` 的前置依赖。
> **For agentic workers:** REQUIRED SUB-SKILL: Use `executing-plans` or equivalent task-by-task workflow.

## 1. 动机

当前 Hive 的文件处理是**用完即丢**：

```
IM 附件 → base64 → fileconv → 喂 LLM → 丢弃
```

问题：
- 用户发的图片/文件无法在后续 turn 引用
- KB 文档中的图片无处持久化
- Agent 生成的图表/报告无法回溯
- 没有统一的 ACL、去重、审计

## 2. 设计原则

- **Provider Pattern**：interface + 多实现，和 `MemoryStore`/`VectorStore`/`CandidateStore` 同构
- **S3 协议兼容**：MinIO 是默认实现，但 interface 不绑定 MinIO API，任何 S3 兼容存储可替换
- **元数据与对象分离**：对象体存 MinIO/S3，元数据（owner、namespace、tags、ACL）存 PostgreSQL
- **content_hash 去重**：同一文件只存一份对象，多个引用指向同一 key
- **签名 URL 访问**：前端不直连 MinIO，通过 API 拿临时签名 URL
- **业务权限外置**：`asset://` 不是授权凭证；AssetService 只做 owner 基础过滤，KB/IM/Agent 生成物的业务 ACL 由调用方传入 resolver 校验。
- **禁止平行资产层**：KB、fileconv、IM、客服都不能各自保存图片/文件对象，只能复用 `internal/asset`。

## 3. 核心模型

### 3.1 ObjectStore interface

```go
// internal/asset/store.go
package asset

import (
    "context"
    "time"
)

type ObjectStore interface {
    Put(ctx context.Context, key string, data []byte, meta ObjectMeta) error
    Get(ctx context.Context, key string) ([]byte, ObjectMeta, error)
    Head(ctx context.Context, key string) (ObjectMeta, error)
    Delete(ctx context.Context, key string) error
    Exists(ctx context.Context, key string) (bool, error)
    SignedURL(ctx context.Context, key string, ttl time.Duration) (string, error)
}

type ObjectMeta struct {
    ContentHash string            `json:"content_hash"`
    MimeType    string            `json:"mime_type"`
    Size        int64             `json:"size"`
    Tags        map[string]string `json:"tags,omitempty"`
}
```

### 3.2 AssetRecord（PostgreSQL 元数据）

```go
// internal/asset/types.go
type AssetRecord struct {
    ID          string    `json:"id"`           // UUID
    Key         string    `json:"key"`          // MinIO object key: {namespace}/{content_hash}.{ext}
    Namespace   string    `json:"namespace"`    // "im/feishu/tenant_abc" | "kb/refund_policy/doc-001"
    ContentHash string    `json:"content_hash"` // SHA-256
    MimeType    string    `json:"mime_type"`
    Filename    string    `json:"filename"`     // 原始文件名
    Size        int64     `json:"size"`
    OwnerScope  string    `json:"owner_scope"`  // "user" | "tenant" | "system"
    OwnerID     string    `json:"owner_id"`
    Tags        map[string]string `json:"tags,omitempty"`
    CreatedAt   time.Time `json:"created_at"`
}

// AssetURI 统一资源标识
// 格式: asset://{namespace}/{content_hash}.{ext}
type AssetURI string

func (u AssetURI) ToObjectKey() string // 转成 MinIO key
func NewAssetURI(namespace, contentHash, ext string) AssetURI
```

URI 约束：

- `asset://` 是 opaque internal URI，不是公开 URL，也不能直接作为 `<img src>`。
- namespace 必须是逻辑隔离路径，不要只用业务名。KB 图片建议使用 `kb/{owner_scope}/{owner_id}/{kb_namespace_id}/{document_id}`，避免跨 owner 同名 namespace 混淆。
- URI 中不能包含 secret、bucket、真实本地路径或 provider 细节。
- object key 可以继续等于 `{namespace}/{content_hash}.{ext}`；如果未来做全局对象体去重，可以在 ObjectStore 层共享对象，但 PG metadata 必须 owner-scoped。

### 3.4 UploadOpts

```go
type UploadOpts struct {
    Namespace  string // 必填
    Filename   string
    MimeType   string
    OwnerScope string
    OwnerID    string
    Tags       map[string]string
}
```

业务 ACL 扩展：

```go
type ResolveContext struct {
    OwnerScope string
    OwnerID string
    UserID string
    SessionID string
    DomainID string
    Purpose string // "chat_attachment" | "kb_document_image" | "agent_artifact"
    Extra map[string]string
}

type AccessResolver interface {
    CanResolveAsset(ctx context.Context, rec *AssetRecord, rc ResolveContext) error
}
```

`AssetService.GetSignedURL` 不应裸签任意 key。API 层必须先查 `AssetRecord`，再执行 owner 和业务 resolver 校验；KB 图片由 KB resolver 校验 document/node/binding 权限。

### Phase 3：S3 兼容实现

| 文件 | 职责 |
|---|---|
| `internal/asset/s3_store.go` | AWS S3 / 阿里云 OSS 实现 |

实际上 `minio-go/v7` 本身就是 S3 兼容客户端。当前实现已合并为同一个 S3-compatible store，区别只在：
- endpoint 默认值（MinIO 是显式地址，S3 是 region-based）
- use_ssl 默认值（MinIO 默认 false，S3 默认 true；S3-compatible HTTP endpoint 可显式设置 `use_ssl=false`，Normalize 不会覆盖）
- 认证方式（MinIO 用 access_key，AWS 可以走 IAM role）

实现层通过 `asset.provider=minio|s3` 做配置区分，不维护第二套对象存储代码。

### Phase 4：KB 集成（由 KB P2 计划消费）

本 phase 由 `KB P2 资产、PDF/DOCX 与客服试点` 触发：

- KB ingest 时提取图片，调用 `AssetService.Upload(namespace="kb/{owner_scope}/{owner_id}/{kb_namespace_id}/{document_id}")`。
- 上传 tags 必须包含 `source_kind=kb_document_image`、`kb_namespace_id`、`kb_document_id`、`domain_id`，node 归属确定后补 `kb_node_id` 或写 `kb_node_assets` 关系表。
- markdown 中本地路径或 data URI 替换为 `asset://...`，禁止把 base64 或本地相对路径留在 `kb_tree_nodes.text`。
- `kb.section.text` 返回结构化 `asset_refs`；`<kb_asset>` 只可作为展示 sentinel，不能替代服务端结构化字段。
- `GET /api/v1/assets/resolve?uri=asset://...` 对 `source_kind=kb_document_image` 必须调用 KB 权限校验：管理态仅允许同 owner 的 `purpose=kb_management`，运行态必须带 `session_id/domain_id` 并命中有效 KB binding 后才签发 URL。
- signed URL TTL 对 KB 图片默认不超过 10 分钟；response 不返回 bucket、secret、object key 内部细节。

## 6. 不做什么

- 不做 CDN 加速（签名 URL 直连 MinIO，后续按需加 CDN）
- 不做图片处理（缩略图、裁剪、水印——后续按需加）
- 不做版本管理（MinIO 支持 versioning，但第一版不开）
- 不做跨 bucket（单 bucket `hive-assets`，namespace 做逻辑隔离）
- 不做流式上传（第一版全量 `[]byte`，大文件场景后续加 multipart）
- 不在 AssetService 内实现 KB binding、IM 会话权限或 Agent artifact 策略；这些由业务 resolver 接入。
- 不允许调用方把对象体、base64 或本地路径存进业务文本表来绕过资产层。

## 8. 风险

| 风险 | 缓解 |
|---|---|
| MinIO 单点故障 | 生产用 MinIO cluster 或直接对接云 S3 |
| 大文件 OOM | `AssetService.Upload` 限制单文件 100MB；聊天附件入口仍保留 25MB 单附件上限；multipart 后续加 |
| 签名 URL 泄漏 | 默认 TTL 短时化；KB 图片不超过 10 分钟；前端不缓存 URL；HTTPS only |
| 元数据和对象不一致 | Upload 先写对象再写 PG；PG 保存失败时会删除本次新写对象，已存在共享对象不删除；仍可通过 Admin GC 清理历史孤儿对象 |
| `asset://` 被当作 ACL | API resolve 必须查 metadata 并调用 owner/业务 resolver；直接按 URI 签名是安全漏洞 |
| KB/IM 重复造资产层 | 计划和测试要求所有文件对象走 `internal/asset`，业务包只保存关系和引用 |

本轮已落地 `internal/asset` 的可运行核心，并补齐 MinIO / S3-compatible provider、对象一致性保护、附件链路和 Agent artifact 基础持久化：

- `internal/asset` 已提供 `ObjectStore`、`ObjectLister`、`AssetMetaStore`、`AssetService`、`AssetURI`、owner-scoped dedupe、local object store、S3-compatible object store、PG metadata store、access resolver 接口。
- `internal/store/postgres_migrate.go` 已新增 `assets` metadata 表，并作为 schema 单一权威；asset bootstrap 复用已完成的 PostgresStore migration，不再维护第二份 asset DDL。
- `internal/bootstrap/asset.go` 已按 config 初始化 `local` / `minio` / `s3` provider 和 PG metadata store，不再把非 local provider 降级为 local。
- `docker-compose.yml` 已新增 MinIO 服务和 `minio-init` bucket 初始化任务；本地已启动 `hive-minio`，并创建 `hive-assets` bucket。`docker/config.docker.json` 默认使用 `asset.provider=minio`，生产可改 `asset.provider=s3` 接 AWS S3、百度 BOS 等 S3-compatible 对象存储。
- HTTP 已新增 `GET /api/v1/assets/resolve`、`GET /api/v1/assets/proxy` 和 Admin `POST /api/v1/admin/assets/gc`。resolve 不接受 query 传入 owner；认证开启时从当前用户派生 `owner_scope=user` / `owner_id=user.ID`，认证关闭的本地模式使用 `owner_id=local`。
- `ResolveAsset` 和 HTTP resolve 都改为 fail closed：没有显式 `AccessResolver` 时拒绝；默认 bootstrap resolver 只允许当前用户解析自己的非 KB user 资产。
- `source_kind=kb_document_image` 已接入 KB resolver 校验 owner/domain/binding/document/node；KB 管理态要求同 owner user，聊天运行态使用 `purpose=kb_section_text` 且要求有效 KB binding。
- `source_kind=agent_artifact` 和 `source_kind=chat_attachment` 已接入同 owner user 校验，并强制要求请求 `session_id` 和 asset metadata 中的 `session_id` 匹配；认证关闭的本地会话使用 `owner_id=local` 持久化和解析本地会话资产。
- local provider 不再向前端暴露 `file://` 绝对路径；resolve 会把 local URL 显式解析为 `asset://` object key 后改写成短时同源 proxy URL，proxy 校验签名和过期时间后流式返回对象，避免把 `local://asset/...?...` 的 query 混入对象 key。
- KB PDF/DOCX/Markdown 图片链路复用本对象存储层：Admin/API 上传入口只接受 `multipart/form-data`，文档文件字段为 `file`，图片字段为重复 `assets`；`internal/fileconv` provider 产出的图片资产和同批上传图片都会先进入 `internal/kb/asset_ingest.go`，再统一上传为 `asset://kb/...`；不得在 fileconv、KB 或客服域保存第二套本地文件路径。系统未上线，无老客户端兼容负担；JSON/base64 ingest API 已删除，非 multipart 请求返回 415。
- Web 聊天附件在验证通过后先写入 `AssetService`，再继续复用现有 `fileconv.Convert` 进入 LLM；`FileAttachment` 已补齐 `asset_uri`、`content_hash`、`size` 字段，前端历史消息展示必须以 `purpose=chat_attachment` 和当前 `session_id` resolve。
- 飞书 IM 入站附件通过 `channel.AttachmentDownloader` 抽象下载为标准附件，Router 不绑定飞书实现；Master 统一上传为 `source_kind=chat_attachment` 后继续走现有附件/文档转换路径。
- 微信 wechatbot 入站图片/文件/视频/语音已复用官方 SDK 下载能力接入同一 `AttachmentDownloader` 抽象；含附件消息跳过 sender debounce，避免消息合并后丢失原始媒体上下文。
- Assistant 输出的已闭合 `<artifact>` 会在 `persistAssistant` 前上传到 `AssetService`，消息 metadata 和 WebSocket payload 携带 artifact manifest；前端 Canvas 仍兼容旧文本解析，并可按 manifest 懒加载资产内容。
- `AssetService.Upload` 已增加 100MB 对象上限；对象写成功但 PG metadata 保存失败时，会补偿删除本次新对象，且不会删除已存在的共享 content-addressed 对象。
- local / MinIO / S3-compatible provider 均支持 `ListKeys`，`AssetService.GCOrphanObjects` 可 dry-run 或删除缺失 PG metadata 的孤儿对象。
