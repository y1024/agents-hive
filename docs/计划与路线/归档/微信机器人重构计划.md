# 微信个人 Agent 通道重构计划：wechatbot.dev + 用户级会话闭环

> 重写日期：2026-05-10
> 状态：COMPLETED / ARCHIVED（2026-05-11）。本文件取代旧版 `微信机器人重构计划.md`。
>
> **归档记录（2026-05-11）：** 本计划已按官方 wechatbot.dev / iLink 通道完成当前范围闭环并归档。已落地官方 SDK 对齐、旧 WeChatPadPro / Wechaty 产品入口清理、用户级扫码连接、IM session 隔离、`ReplyToken` 回复链路、context token 持久化、typing/媒体安全元数据、Router owner scope、pending input/HITL 渲染，以及定时任务跨重启主动发送的 `no_context` 边界。当前产品边界明确为微信里的 `clawbot` 联系人通道，不承诺接管扫码微信号 A 的普通私聊，也不承诺群聊能力。
>
> **已锁定的关键决策（2026-05-10）：**
> 1. **产品定位**：销售 / 老板个人助手——付费个人或企业采购后分发个人使用。MVP 真实场景是"1:1 客户咨询补充"。企业全员场景走企微通道。
> 2. **企微平行**：本计划范围不变（个人微信）；企微通道补齐作为**平行优先项**，单独立项。
> 3. **Phase -1 SDK Spike**：先实测官方 wechatbot.dev SDK，再删除旧协议。Spike 不通过时暂停计划，不能先把回退路径删掉。
> 4. **Phase 0 旧协议**：Spike 通过后硬删除 `wechatpadpro/` `wechaty/`，grep gate 严格执行。**接受的代价**：one-way door，wechatbot.dev SLA 出问题无回退路径，必须靠 Phase 4 上线 hardening + canary 兜底。
> 5. **统一 IM scope 契约**：不得为 wechatbot 开绕过 Router 的 Master / tools / session 发送专线。wechatbot 必须作为 `ChannelPlugin` 接入统一 Router；入站用 `channel.InboundMessage.OwnerUserID`，出站跨包 DTO 用 `imctx.SendRequest.OwnerUserID`，插件发送用 `channel.OutboundMessage.OwnerUserID`。`TenantKey=OwnerUserID` 参与 `BuildSessionID`。Router 通过 `OwnerUserResolver` 注入 auth user；禁止用 context value 传 owner，禁止修改 `enrichCtx` 签名。
>
> **群聊能力硬事实（当前 SDK 证据）：**
> - 不能从“网上说 iLink 底层支持群聊”直接推导出“当前 pinned wechatbot Go SDK 可稳定接入群聊”
> - 当前官方 Go SDK / iLink 公开文档未暴露稳定群聊 RoomID/ChatroomID/GroupID 入口
> - 当前发送模型围绕 `userID` 与 prior `context_token`，未提供稳定 SendToRoom API
> - 因此不要在 MVP 承诺群聊、@、群成员管理或任意群推送
> - 因此 MVP 仅支持 1:1 联系人会话，群聊延后到上游 SDK 支持后另议
>
> **2026-05-11 关键纠偏：官方 wechatbot.dev / iLink 支持微信里的 clawbot 自动回复，但不是个人微信号 A 的普通私聊托管。**
> - SDK 登录走 `/ilink/bot/get_bot_qrcode?bot_type=3`，成功后得到的是 `bot_token` / `ilink_bot_id` / `ilink_user_id`。
> - SDK 长轮询走 `/ilink/bot/getupdates`，`parseMessage` 只把 `message_type == user` 的 Bot 收件箱消息抛给 `OnMessage`；客户 B 给微信里的 `clawbot` 发消息后，可以触发 `OnMessage` 并用 `Reply` 自动回复 B。
> - 用户扫码后在微信里看到的 `clawbot` 才是消息入口；外部联系人 B 给扫码微信号 A 本人发普通私聊，不会进入 SDK `OnMessage`，本系统也无法处理。
> - 因此本计划必须从“个人微信账号普通私聊自动托管”降级为“官方 wechatbot.dev/iLink 支撑的 clawbot 联系人通道”。如产品目标是客户给员工个人微信发消息后自动接管，当前官方 wechatbot.dev SDK 不能满足。
>
> **2026-05-11 SDK 对齐执行计划：** `docs/superpowers/plans/2026-05-11-wechatbot-sdk-alignment.md`。该计划锁定 `ReplyToken` 贯穿 `InboundMessage -> Router -> OutboundMessage -> wechatbot.Plugin -> SDK Reply`，并补齐 typing、安全媒体元数据、context token 持久化和 debounce scope 修复。完整媒体下载 / 预览仍需独立附件存储方案。
>
> **自动回复与密钥结论：** SDK/协议没有额外 `auto_reply` 开关，也不要求手工配置 `app_secret` / webhook secret。必须开启的是本仓库 `channel.wechatbot.enabled`，用户必须扫码登录；扫码确认后 SDK 获得并保存 `bot_token`，后续回复还依赖每条入站消息携带的 `context_token`。
>
> **2026-05-11 SDK 对齐实现决策：**
> - 直接回复：使用 `InboundMessage.ReplyToken -> OutboundMessage.ReplyToken -> Backend.Reply`。
> - 同进程主动发：优先 SDK `Send`，使用 SDK 内部 `sync.Map` prior context。
> - 跨重启主动发 / 定时任务：使用本仓库加密持久化的 `context_token` 走 `SendWithContextToken`；没有上下文时返回 `no_context`。
> - 媒体入站：仅保存安全元数据；不得把 CDN media、AES key、context token、raw wire message 暴露给 Web API。
>
> **2026-05-14 IM 统一抽象收口补充：** 微信主动外发已纳入统一 `im_api -> imcore.Service -> WechatBotAdapter` 链路。`send_im_message` 仍保留为 legacy 兼容入口，但内部同样委托 `imcore.Service`；模型不能通过参数传入 `owner_user_id`，owner 只能从当前 auth context 派生。`WechatBotAdapter` 必须按 owner 校验 conversation/peer wxid 归属和 `CanSend`，没有明确 `recipient_id` / `conversation_id` 时必须先让用户确认目标，即使只有一个可发送会话也不能自动选择。

## 0. 结论先行

这次重构不能再被定义为“把个人微信号接入 Agent”。基于 2026-05-11 对官方 SDK 源码的复核，wechatbot.dev 当前交付的是 **官方 wechatbot.dev/iLink 支撑的 clawbot 联系人通道**，不是扫码微信号 A 的普通私聊监听器。

可成立的闭环是：

1. 每个登录用户扫码连接微信里的 `clawbot`，底层由官方 wechatbot.dev/iLink 提供凭证与长轮询。
2. 外部联系人必须向微信里的 `clawbot` 发消息，消息才会进入该用户自己的 Agent 会话。
3. Web 端只管理连接状态和最近联系人状态，不展示 Bot 消息内容，也不提供普通 Chat 接管回复。
4. 系统能清楚解释“未登录、已掉线、跨用户访问”等失败原因。
5. WeChatPadPro (iPad) 和 Wechaty (gRPC) 必须从产品与代码路径中清空；个人微信只允许走官方 wechatbot.dev / iLink SDK。

MVP 的关键不再是“客户给员工个人微信发消息自动托管”，而是“用户拥有一个官方 Bot 联系人，客户进入 Bot 后由 Agent 在 IM 内处理回复，Web 只做连接与状态管理”。如果业务必须接管客户发给员工个人微信号 A 的普通私聊，本计划应暂停，转向企业微信/微信客服/公众号客服等官方企业或公众平台入口。

## 1. 旧计划必须修正的问题

| 旧计划假设 | 当前问题 | 新计划修正 |
| --- | --- | --- |
| `enrichCtx(ctx, wxid, "wechatbot")` 可把微信用户映射成系统用户 | `wxid` 是外部联系人，不是 agents-hive 登录用户；这会把外部联系人错当系统用户 | 系统用户来自 `BotInstance.OwnerUserID`，外部联系人只作为 `PeerWxid` |
| Session ID 写成 `im-wechatbot-{wxid}` | 当前 IM 会话必须通过 `imctx.BuildSessionID(platform, tenantKey, chatID)` 生成 | 新增 `PlatformWeChatBot = "wechatbot"`，使用 `BuildSessionID("wechatbot", ownerUserID, peerWxid)` |
| `/api/wechat/*` | 当前 API 统一使用 `/api/v1/*` | 新接口统一放在 `/api/v1/wechat/*` |
| WeChat 设置页还需要新建 | `frontend/src/pages/Settings.tsx` 已存在，但尚未在 `App.tsx` 注册路由 | 复用并扩展 `/settings`，不要重新发明设置页 |
| IM session 会出现在普通会话列表 | 普通 Web 会话面与 IM 入站 session 混在一起会泄露渠道上下文，也与飞书处理逻辑不一致 | 保持所有 `im-*` session 对普通 Web 会话列表不可见，微信与飞书一致 |
| 普通 Web Chat 代发微信天然可用 | `POST /api/v1/sessions/{id}/messages` 是普通 Web Chat 入口，不应承担 IM 代发 | 不提供普通 Web Chat 代发；IM 回复只走 IM renderer / Router 内部链路 |
| `send_im_message` 可以直接支持用户级微信发送 | `internal/tools/sendim.go` 的 `SendMessage(ctx, platform, chatID, content)` 缺少 scope 维度 | 工具层走统一 IM send request；wechatbot 的 owner 来自 auth context，禁止让 Agent 传任意 `owner_user_id` |
| 保留 WeChatPadPro / Wechaty 作为回退 | 当前实现是非官方协议，会继续污染配置、工具、前端和路由心智 | Phase 0 先清空旧协议入口与实现，个人微信唯一官方路径是 wechatbot.dev |
| 数据库使用 `UUID REFERENCES auth.users(id)` | 当前仓库用户表是 `users(id TEXT)`，没有 `auth.users` schema | 新迁移必须使用 `TEXT REFERENCES users(id)` |
| 任意主动发到任意 wxid | wechatbot Go SDK 的 `Send` / `SendMedia` 需要 prior context；冷启动推送不是 MVP 能承诺的能力 | 产品文案和接口都明确“仅可发送到已有上下文的会话” |
| B 给扫码账号 A 发普通微信私聊会进入系统 | SDK 接入的是官方 iLink Bot 收件箱；A 的普通微信私聊不会进入 `/ilink/bot/getupdates` | 验收路径必须改成“B 给微信里的 `clawbot` 发消息” |

## 3. 产品北极星

### 3.1 目标用户

目标用户不是系统管理员，而是每一个已登录 agents-hive 的普通用户；但这里的“微信连接”不是托管他的微信号普通私聊，而是给他开通一个由官方 wechatbot.dev/iLink 支撑的 `clawbot` 联系人入口：

- 他希望把微信里的 `clawbot` 联系人发给客户，让客户可以直接找他的 Agent。
- 他希望在 Web 端确认连接状态和最近联系人状态。
- 他不应在普通 Web Chat 中看到或接管 IM 消息；IM 消息回复由 Agent/IM renderer 在渠道内完成。
- 他不应该理解 WeChatPadPro、Wechaty、tenant、chatID、context token 这些实现细节。

### 3.2 第一版 10 分体验

1. 用户打开 `/settings`。
2. 点击“连接微信”，看到二维码。
3. 手机微信扫码并确认。
4. 页面显示 clawbot 连接在线状态，以及明确说明“请让对方给微信里的 clawbot 发消息”。
5. 某个微信联系人给 `clawbot` 发消息。
6. Agent 在微信 Bot 会话内处理并回复。
7. Web 设置页最近联系人状态更新，但左侧普通会话列表不出现微信 Bot 会话。
8. 用户手动访问 `/sessions/im-wechatbot-*` 不会看到消息内容。

### 3.3 成功指标

| 指标 | MVP 目标 |
| --- | --- |
| 扫码成功到在线状态 | 30 秒内完成，失败给出可操作错误 |
| 微信 Bot 入站到设置页最近联系人状态更新 | 5 秒内可见 |
| 同一联系人会话稳定性 | 同一 owner + peer 总是落到同一 session |
| 普通 Web Chat 隔离 | 0 容忍：`im-*` 不进入会话列表、详情、消息、WebSocket |
| 跨用户隔离 | 0 容忍：用户 A 不能看到或操作用户 B 的微信连接状态 |
| 官方协议纯度 | 代码与 UI 不再提供 WeChatPadPro / Wechaty 入口 |

## 4. 官方 SDK 约束

参考：

- GitHub: `https://github.com/corespeed-io/wechatbot`
- Go SDK 文档: `https://www.wechatbot.dev/zh/golang`
- iLink 协议文档: `https://www.wechatbot.dev/zh/protocol`

截至 2026-05-10 核对，Go SDK 关键能力如下：

| 能力 | 对产品/工程的约束 |
| --- | --- |
| `go get github.com/corespeed-io/wechatbot/golang` | 依赖升级要检查当前 `go.mod` 与构建环境 |
| `bot.Login(ctx, force)` 返回 credentials，包含账号身份 | 登录成功后用 SDK 返回的账号 ID 作为 `OwnerAccountID` |
| `Options` 支持 `CredPath`、`OnQRURL`、`OnScanned`、`OnExpired`、`OnError` | 后端需要事件桥接到 WebSocket / SSE |
| 二维码登录 | 不需要手填 wechatbot app secret；确认登录后 SDK 保存 `bot_token` 到 `CredPath`，这是后续 iLink API 的 Bearer token |
| `bot.OnMessage(func(*IncomingMessage))` | 入站消息由 BotInstance 转成内部会话消息 |
| `bot.Reply(ctx, msg, text)` | 客户 B 给微信里的 `clawbot` 发消息后的自动回复必须优先走 Reply，直接使用该条入站消息的 `ContextToken` |
| `bot.Send(ctx, userID, text)` | 只使用 SDK 内部 `sync.Map` 中的 prior context；适合同进程已有上下文的主动发送，不适合承诺跨重启定时任务 |
| `context_token` | 协议要求每次发送都带 token；官方 Go SDK 只在内存 `sync.Map` 缓存，仓库若要支持跨重启定时/主动发送，必须自行加密持久化 token 并在 session expiry / relogin / logout 时清理 |
| `SendTyping` / `StopTyping` | SDK 已支持输入中状态；当前 adapter 需要补暴露并在入站处理期间 best-effort 调用 |
| `ReplyContent` / `SendMedia` / `Download` | SDK 已支持图片、文件、语音、视频的回复、主动发送和下载；MVP 可先保留媒体元数据，完整下载/预览必须走独立附件存储，不得把 CDN 密钥暴露给 Web |
| Go SDK 与顶层/Node 文档差异 | 顶层 README 写所有 SDK 共享 context token 跨重启持久化，但当前 pinned Go SDK 只有内存 `sync.Map`；Node SDK 有 `ContextStore` 持久化。实现必须以 pinned Go module 为准，不把 Node 行为套给 Go。 |
| 协议版本差异 | 当前 pinned Go SDK hardcode `ChannelVersion="0.1.0"` / `iLink-App-ClientVersion="256"`；Node SDK 从 package version 读取，fallback `2.0.0`。实施前必须确认是否升级 Go SDK，避免协议版本行为和文档不一致。 |
| 会话过期 `-14` 自动触发重新登录 | 产品状态需要区分 `online`、`relogin_required`、`recovering`、`offline` |
| **群聊** | iLink 底层可能存在群聊能力，但当前 pinned 官方 Go SDK / 公开协议没有暴露可稳定依赖的群聊收发模型。MVP 仅 1:1 联系人；群聊必须等真实 SDK Spike 或上游文档确认后另议。 |
| **@ 提及 / 消息撤回 / 群成员管理** | 当前官方文档未提供可依赖能力，MVP 不承诺。 |
| **个人微信普通私聊监听** | 当前 SDK 以官方 iLink Bot 身份登录、长轮询 Bot 收件箱；不监听扫码账号 A 的普通微信私聊。B 给 A 发消息不会触发 `OnMessage`。 |

### 4.1 运行前置条件

必须同时满足：

1. 管理端或配置开启 `channel.wechatbot.enabled=true`。这是本仓库的官方微信 Bot 通道开关，不是 SDK 自动回复开关。
2. 普通用户在 `/settings` 扫码登录。SDK 通过 `/ilink/bot/get_bot_qrcode?bot_type=3` 和 `/ilink/bot/get_qrcode_status` 获取 `bot_token` / `ilink_bot_id` / `ilink_user_id`。
3. 客户 B 给微信里的 `clawbot` 发消息。只有这类消息进入 `/ilink/bot/getupdates`。
4. Router/Master/Agent 正常处理并生成回复。LLM API key、模型配置等沿用系统现有配置，不属于 wechatbot SDK 密钥。
5. 回复时必须有 `context_token`。直接回复使用本次入站消息的 token；主动/定时发送需要 prior token，跨重启必须持久化。

明确不需要：

- 不需要配置 `app_secret`、回调签名 secret 或 webhook token。
- 不需要开启 SDK 级“自动回复”开关；自动回复来自 `OnMessage -> Router/Master -> Reply` 这条代码链路。
- 不需要授权读取扫码微信号 A 的普通个人私聊；SDK 不读取这类私聊。

## 5. 身份模型

这是整个重构的核心。任何实现绕开它，都会造成串号、越权或无法回发。

### 5.1 三个身份

| 字段 | 含义 | 示例 | 归属 |
| --- | --- | --- | --- |
| `OwnerUserID` | agents-hive 登录用户 ID | `user_abc` | 系统内部用户 |
| `OwnerAccountID` | 扫码登录成功后的后台 Bot 身份 ID | SDK credentials 中的 `ilink_bot_id` | 用户绑定的官方 clawbot 身份 |
| `PeerWxid` | 给该 Bot 发消息的外部联系人 ID | `wxid_xxx` | 外部联系人 |

必须禁止的混淆：

- `PeerWxid` 不是系统用户 ID。
- `OwnerAccountID` 不是 agents-hive 用户 ID。
- `OwnerUserID` 不能从入站消息 sender 反查，必须来自 BotInstance。

### 5.2 绑定关系

`OwnerUserID` 与 `OwnerAccountID` 是“该系统用户连接了哪个官方后台 Bot 身份字段对应的 clawbot”的绑定。它用于：

- 登录状态展示。
- 凭证目录隔离。
- 防止同一 Bot 账号被多个系统用户重复绑定。
- 入站消息创建 session 时注入系统用户上下文。

`PeerWxid` 与 `OwnerUserID` 是“该用户与哪个外部联系人有会话”的关系。它用于：

- 构造稳定 session。
- 展示联系人昵称、头像、最近消息。
- 判断 Agent/IM renderer 是否有可用发送上下文。

### 6.1 Platform 常量

```go
// internal/channel/types.go
PlatformWeChatBot Platform = "wechatbot"

// internal/imctx/context.go
PlatformWeChatBot Platform = "wechatbot"
```

旧协议常量必须删除：

- `wechat-wechaty`
- `wechat-wechatpadpro`

删除要求：

- `internal/channel/types.go` 不再暴露旧微信 Platform。
- `internal/imctx/context.go` 不再暴露旧微信 Platform。
- `internal/tools/sendim.go` 的 platform enum 不再包含旧微信协议。
- 任何旧协议测试、文案、配置示例同步清理。

### 6.2 Session ID

MVP 采用现有 IM session 规范：

```go
sessionID, err := imctx.BuildSessionID(
    imctx.PlatformWeChatBot,
    ownerUserID,
    peerWxid,
)
```

语义：

- `platform = wechatbot`
- `tenantKey = OwnerUserID`
- `chatID = PeerWxid`

这样同一个外部联系人给不同系统用户的微信号发消息时，会落到不同 session：

```text
im-wechatbot-user_a-wxid_peer
im-wechatbot-user_b-wxid_peer
```

禁止：

```text
im-wechatbot-{wxid}
```

原因：

- 没有 owner 维度，会跨用户串会话。
- 绕过 `BuildSessionID`，违反现有 IM 规范。
- 无法和现有 router / master / access check 一致。

### 6.3 Session 归属

微信入站创建 session 时，必须让 `sessions.user_id = OwnerUserID`。

确定做法：

1. `BotInstance` 收到 SDK `IncomingMessage`。
2. `BotInstance` 构造统一 `channel.InboundMessage`，必须设置 `OwnerUserID=BotInstance.OwnerUserID`、`TenantKey=OwnerUserID`、`ChatID=PeerWxid`、`SenderID=PeerWxid`。
3. `BotInstance` 调用现有 `router.HandleMessage(ctx, msg)`，不得绕过 Router 直连 Master。
4. Router 识别 user-scoped 插件后，用 `OwnerUserResolver` 按 `OwnerUserID` 加载系统用户，调用 `auth.WithUser(ctx, user)`，并写入 `IMContextValue.InternalUserID`。
5. Router 使用 `BuildSessionID(platform, msg.TenantKey, msg.ChatID)` 得到 session，进入 Master 前确保 session owner 被正确写入。

禁止事项：

- 不新增 `HandleMessageWithOwnerContext` 这种旁路入口。
- 不用自定义 context value 携带 owner。
- 不修改 `enrichCtx` 签名。`enrichCtx(ctx, externalID, provider)` 继续只服务飞书 / 钉钉 / 企微这类 tenant-scoped 企业 IM；wechatbot 的 `PeerWxid` 永远不进入 `enrichCtx`。

### 6.4 Binding Key

当前 Router 的内存绑定 key 是 `platform:chatID`。这在 wechatbot 下会串号：同一个 `PeerWxid` 可能同时给用户 A 和用户 B 的个人微信发消息。

确定改造：

- `channel.Binding` 增加 `TenantKey string`。
- `Bind` / `LookupSession` / `Unbind` 的内部 key 统一改为 `platform:tenantKey:chatID`。
- Router 在入站时先归一化 `tenantKey`：wechatbot 必须是 `OwnerUserID`；tenant-scoped 企业 IM 允许继续用平台 tenant/corp/open tenant，空值才回退 `default`。
- 不保留 `platform:chatID` 新写入路径；当前 binding 是进程内内存状态，不需要做持久化迁移。

### 7.1 用户外部身份绑定

旧计划里的 `UUID REFERENCES auth.users(id)` 是错的。当前仓库用户表为 `users(id TEXT)`。

```sql
CREATE TABLE IF NOT EXISTS user_external_ids (
    id BIGSERIAL PRIMARY KEY,
    user_id TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    provider_type TEXT NOT NULL,

CREATE INDEX IF NOT EXISTS idx_user_external_ids_user_provider
```

说明：

职责：

| 模块 | 职责 |
| --- | --- |
| `wechatbot.Backend` | SDK adapter：Login、Run、Stop、Reply、Send、SendTyping、StopTyping、media/download、status |
| `BotInstance` | 一个系统用户的一条微信连接，持有 OwnerUserID / OwnerAccountID / SDK Bot |
| `wechatbot.Plugin` | 实现 `channel.ChannelPlugin`，作为 wechatbot 进入统一 IM Router 的唯一插件入口 |
| `BotRegistry` | 插件内部管理 `map[OwnerUserID]*BotInstance`，只暴露给 `wechatbot.Plugin` 与登录生命周期服务，不给 Master / session API / tools 直连发送 |

### 8.1.1 统一 IM 接口锁定

IM 必须是统一、可插拔接口，而不是为 wechatbot 硬耦合一套链路。

```go
// internal/channel/types.go
type InboundMessage struct {
    Platform    Platform
    TenantKey   string
    OwnerUserID string // 仅 user-scoped IM 必填；tenant-scoped 企业 IM 为空
    ChatID      string
    SenderID    string
    ReplyToken  string // wechatbot=iLink context_token；其他平台保持空
    // ...
}

type OutboundMessage struct {
    Platform    Platform
    TenantKey   string
    OwnerUserID string // wechatbot 插件发送必填；企业 IM 为空
    ChatID      string
    Content     string
    // ...
    ReplyToken  string // 由 Router 从入站透传，插件用它执行官方 Reply 语义
}

// internal/imctx/send_request.go
// tools / API / Router 共用的跨包出站 DTO。放在 imctx，避免 tools -> channel ->
// master -> tools 的 import cycle。
type SendRequest struct {
    Platform    Platform
    TenantKey   string
    OwnerUserID string // wechatbot 必填；企业 IM 为空
    ChatID      string
    Content     string
    MsgType     string
    ReplyTo     string
    ReplyToken  string
}
```

Router 统一契约：

- `HandleMessage(ctx, InboundMessage)` 是唯一入站入口；不新增 wechatbot 专用 handler。
- `Router.SendMessage(ctx, imctx.SendRequest)` 是统一出站入口；Router 在包内转换成 `channel.OutboundMessage` 后调用插件 `Send`。
- `internal/tools/sendim.go` 的窄接口改成 `SendMessage(ctx, imctx.SendRequest)`；工具包不得 import `internal/channel`，避免 import cycle。
- `OwnerUserID != ""` 表示 user-scoped IM。Router 通过 `OwnerUserResolver` 注入 auth user 并跳过 `enrichCtx`。
- `OwnerUserID == ""` 表示 tenant-scoped IM。Router 保持现有 `enrichCtx(ctx, SenderID, provider)` 行为。
- `wechatbot.Plugin.Send` 要求 `msg.OwnerUserID != ""` 且 `msg.TenantKey == msg.OwnerUserID`，再在插件内部按 owner 找 `BotInstance` 发送；飞书 / 钉钉 / 企微忽略 `OwnerUserID` 零值。
- session API / Master / tools 的发送路径只能依赖 Router 的统一入口；不得 import `internal/channel/wechatbot` 后直接调 `BotRegistry.SendByOwner`。
- `/api/v1/wechat/*` 登录生命周期 API 只能依赖 `WeChatConnectionService` 窄接口，不允许把发送方法挂到该接口上。

### 8.2 入站消息流程

```text
wechatbot SDK OnMessage
  -> BotInstance.handleIncoming(ownerUserID, ownerAccountID, msg)
  -> upsert wechat_conversations(ownerUserID, peerWxid, sessionID)
  -> build channel.InboundMessage{OwnerUserID, TenantKey=OwnerUserID, ChatID=PeerWxid, ReplyToken=msg.ContextToken}
  -> Router.HandleMessage(ctx, inbound)
  -> Router.OwnerUserResolver(ownerUserID) + auth.WithUser
  -> Master / channel processor
  -> Agent response
  -> Router builds OutboundMessage{OwnerUserID, TenantKey=OwnerUserID, ChatID=PeerWxid, ReplyToken}
  -> wechatbot.Plugin finds BotInstance
  -> if ReplyToken != "" use SDK Reply / else use SDK Send prior-context path
  -> send reply in IM channel; normal Web Chat stays isolated
```

关键要求：

- `wechatbot.Plugin` 和 `BotInstance` 必须是 owner-aware，但 owner 只能作为统一 IM scope 流经 Router。
- `ReplyToken` 是统一 IM DTO 字段，不是 context value；wechatbot 用它承载 iLink `context_token`，飞书 / 钉钉 / 企微保持空值。
- `channel.mergeBatch` 必须以最后一条消息为基准保留 `TenantKey` / `OwnerUserID` / `ReplyToken` / metadata，只合并 `Content`，否则 wechatbot debounce 后会丢 owner scope。
- 不得把 `PeerWxid` 传给 `enrichCtx` 当系统用户查。
- `Router` 新增 `SetOwnerUserResolver(func(ctx context.Context, userID string) (*auth.User, error))`，bootstrap 注入 `authEngine.GetUserByIDCached`。
- `processMessageImpl` 中 `msg.OwnerUserID != ""` 时：校验 `msg.TenantKey == msg.OwnerUserID`，解析 active user，注入 `auth.WithUser`，设置 `IMContextValue.InternalUserID`，并跳过 `enrichCtx`。
- `processMessageImpl` 中 `msg.OwnerUserID == ""` 时：保持飞书 / 钉钉 / 企微现有 `enrichCtx(ctx, msg.SenderID, provider)` 行为。
- 入站消息创建 session 前必须完成 owner 注入，否则 auth 开启时 session 会变成无主或越权不可访问。
- 保留 router 现有 dedup、debounce、BuildSessionID、renderer、ack、retry_queue 约束。
- Binding key 必须使用 `platform:tenantKey:chatID`，否则 owner 维度会被 `LookupSession` 绕开。

### 8.3 Web 会话隔离

微信 Bot 入站消息可以创建内部 `im-wechatbot-*` session 供 Router、Master、renderer、journal 和审计链路使用，但普通 Web Chat 不展示也不读取这些 session：

```text
GET /api/v1/sessions
  -> skip all session_id starts with "im-"

GET /api/v1/sessions/{im-*}
GET /api/v1/sessions/{im-*}/messages
POST /api/v1/sessions/{im-*}/messages
GET /api/v1/sessions/{im-*}/journal|trace|todos|trajectory
  -> 404

WebSocket ?session_id=im-*
  -> reject subscription
```

这条规则适用于飞书、钉钉、企微、微信 Bot 等所有 IM 平台。`/api/v1/wechat/conversations` 只在设置页显示最近联系人状态，不跳转普通 Chat，也不承担消息查看或代发。该接口不得返回 `session_id`、`last_message_preview`、`can_send`、`send_state` 等可用于读取消息或接管发送的字段。

### 8.4 主动工具发送

`send_im_message` 不能继续只靠 `platform + chat_id` 发送微信消息。

**改造方向（v0.4 锁定：统一 `imctx.SendRequest`，不暴露 owner 参数给 Agent）：**

```go
type IMRouter interface {
    SendMessage(ctx context.Context, req imctx.SendRequest) error
}
```

调用约定：

- `send_im_message` 的 JSON schema 仍只允许 `platform/chat_id/content`，不增加 `owner_user_id` 字段。
- 工具 handler 从 `auth.UserFrom(ctx)` 派生当前用户。
- 当 `platform=wechatbot`：handler 查 `wechat_conversations(session/chat_id, currentUserID)` 后填 `imctx.SendRequest.OwnerUserID=currentUserID` 与 `TenantKey=currentUserID`。
- 当 `platform=feishu/dingtalk/wecom`：handler 保持 `OwnerUserID=""`，按平台现有 `TenantKey/chatID` 发送。

工具层要求：

- 从 tool context / auth context 解析当前系统用户。
- 禁止 Agent 传入任意 `owner_user_id`。
- 对 `wechatbot` 平台，必须验证 `wechat_conversations.owner_user_id = 当前用户`。
- 无 context token 时返回明确 unsupported/temporarily unavailable，不要静默失败。
- 现有 6 个 WeChat ops 工具在 wechatbot 下先返回“不支持该协议”，避免误导用户。

## 9. API 设计

所有接口均为登录用户级接口，路径统一使用 `/api/v1/wechat/*`。

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| `GET` | `/api/v1/wechat/status` | 当前用户微信连接状态 |
| `POST` | `/api/v1/wechat/login` | 创建/刷新扫码登录流程 |
| `GET` | `/api/v1/wechat/events` | 登录二维码、扫码、过期、在线/掉线事件流 |

### 9.1 状态响应

### 9.2 事件流安全

浏览器 WebSocket 通常不能设置 `Authorization` header。当前仓库 `/api/v1/ws` 已有特殊 token 校验思路；新的 `/api/v1/wechat/events` 必须复用同等级校验，不允许因为是 WebSocket / SSE 就绕过 auth。

- token 放 query 参数时，日志必须脱敏。
- 断线重连不应重复创建多个 BotInstance。

## 12. 安全与隐私

### 12.1 权限边界

- 所有 `/api/v1/wechat/*` 接口必须要求登录。
- Admin 不默认拥有普通用户微信会话访问权；沿用当前 session owner 模型。
- 普通 Web session API 必须拒绝 `im-*`，不能只依赖 `session.user_id == 当前用户` 放行。
- Tool 出站必须从 context 解析当前用户，不能让模型自报 owner。
- Event hub 必须按 user 分区。

### 12.2 日志与指标

禁止：

- 打印二维码原始登录凭证。
- 打印 credentials。
- 在 metrics label 中使用原始 `wxid`、`user_id`、消息内容。
- 在错误里返回 SDK token / context token。

允许：

- 日志中使用 session_id。
- 对 `wxid` 做 hash 后用于 debug。
- 记录 `content_len`，不记录全文。

### 12.3 威胁模型测试

必须覆盖：

- 用户 A 不能订阅用户 B 的 QR 事件。
- 用户 A 不能通过 session id 给用户 B 的 PeerWxid 发消息。
- 同一个 `OwnerAccountID` 不能被两个 `OwnerUserID` 同时绑定。
- BotInstance panic 不影响其他用户 Bot。
- logout 不清理其他用户凭证目录。

1. 用户 A 扫码后，外部联系人 B 给 A 本人的普通微信私聊发消息。
2. 系统不产生会话、不触发 Agent、不回复。
3. 该结果符合当前官方 SDK 边界；UI 和文档必须引导用户给微信里的 `clawbot` 发消息。

## 17. 非目标

本计划明确不做：

- 把微信外部联系人注册成 agents-hive 用户。
- 让管理员代替普通用户扫码。
- 监听或托管扫码微信号 A 的普通私聊。
- 给任意 wxid 冷启动推送。
- **完整群聊支持（当前 pinned SDK / 公开协议未暴露稳定群聊 API，需真实 SDK Spike 或上游文档确认后另议）**。
- **企业全员场景（当前微信 Bot 链路不能承诺群聊覆盖；走企微通道平行立项）**。
- **@ 提及 / 消息撤回 / 群成员管理 / 朋友圈 / 好友管理 / 多账号绑定 / 端到端加密**（SDK 均不支持）。
- 把 WeChatPadPro / Wechaty 作为官方链路失败后的回退方案。
- 改造整个 auth provider 模型。
- 顺手开放所有 IM session 到前端。

## 18. 最终产品判断

这次重构的产品价值不在“支持了新的微信 SDK”，而在“官方 Bot 会话变成用户自己的 Agent 会话”。它不是个人微信普通私聊托管方案。

因此优先级必须保持：

1. 身份与会话归属正确。
2. 用户扫码和状态反馈清楚。
3. 官方 Bot 入站后设置页最近联系人状态可见。
4. 普通 Web Chat 不展示、不读取、不订阅 IM session。
5. 工具和媒体逐步增强。
6. SDK 对齐修正优先于媒体增强：先完成 `ReplyToken`、debounce scope、typing 和 context token 持久化边界。
7. 旧协议在 Phase 0 清空，后续不再回头兼容。

只要 `OwnerUserID / OwnerAccountID / PeerWxid` 三者边界不乱，后续功能都能自然增长；一旦这三个身份混在一起，系统会在权限、会话、发送和审计上同时失控。

### 跨计划集成点（v0.2 新增）

- **Memory RuntimeContext**：wechatbot 入站可以提供 `TaskType="im_wechat_chat"` 这类 IM 上下文，但当前 Master 会在 ReAct 入口重写 `memoryRuntimeContext(..., "react")`。因此要么让 Master merge 现有 RuntimeContext，要么把该集成列为后续增强，不能只靠 BotInstance 塞 context。
- **Multi-agent 工作台**：微信入站默认走 Master ReAct（轻路径），不进 Planner。除非用户消息触发"做个 X"等关键词由 Master 路由器判定为重任务。
- **企微平行计划**：本计划完成后启动企微通道补齐（群聊、合规、企业场景）。两条通道共享统一 IM Router、tenant-aware binding key 与 session 展示策略，不共享个人微信 owner 语义。

### Phase -1 SDK Spike（强制前置）

Phase 0 启动前必须做 1-day spike；截至 2026-05-11，源码复核已经补充如下事实：Go SDK `OnMessage` 顺序调用 handlers，SDK 内部 context cache 是 `sync.Map`，`Reply` 使用入站 `ContextToken`，`Send` 只读内存 prior token，协议要求 token 每次发送必带。

剩余真实环境 spike：
1. 拉 `github.com/corespeed-io/wechatbot/golang` 跑 hello world
2. 实测扫码登录耗时（计划承诺 30s 内）
3. 实测 `Send` 在无 context_token 时的具体错误码
4. 实测 `OnMessage` 回调的 goroutine 模型（是否 SDK 内单 goroutine 串行？panic 隔离？）
5. 实测多用户多 BotInstance 并发是否有共享状态污染

Spike 通过后再进 Phase 0；Spike 失败则计划暂停，重新评估。
